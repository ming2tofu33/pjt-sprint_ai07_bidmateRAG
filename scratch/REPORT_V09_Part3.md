## 6.1 시나리오 B (상용 API) 정량 성능 종합 분석

### 실험 환경 및 측정 조건

시나리오 B의 정량적 성능 평가는 GPT-5-mini 모델을 기준으로, 80문항 평가셋(Evaluation Set)에 대해 반복 실험을 수행하여 측정하였다. 모든 실험은 YAML 설정 파일 1장 정의 → CLI 1줄 실행 → 자동 로깅의 파이프라인을 통해 재현 가능한 형태로 기록되었다.

#### 핵심 발견 3선

**발견 1: Judge v2 도입이 야기한 "지표 정상화 이벤트"**

bench-85377755부터 bench-db62df5d까지 4회 연속 MRR = 0.000이 기록된 것은 RAG 파이프라인의 검색 실패가 아니라, **LLM Judge 평가 로직의 버그**로 인해 정답 판정이 전혀 이루어지지 않았기 때문이다. `feat: add LLM judge v2 with evidence-based ratio scores` 커밋이 적용된 bench-pr-test-v1에서 MRR이 즉시 1.000으로 정상화되었다. 이는 평가 엔진의 신뢰성이 곧 성능 지표의 신뢰성임을 극명하게 보여주는 사례다.

**발견 2: Top-K 증가의 수확 체감 법칙 (Diminishing Returns)**

Top-K를 5 → 30 → 40 → 50으로 확대한 실험에서, MRR은 0.988로 사실상 동일하게 유지된 반면, Latency와 Cost는 선형 이상으로 증가하였다.

| Top-K | MRR | Latency (ms) | Cost (USD) | MRR 대비 Cost 효율 |
|---|---|---|---|---|
| 5 | 1.000 | 5,305 | 0.001 | 최적 |
| 30 | 0.988 | 17,263 | 0.354 | △354배 비용 증가, MRR 오히려 0.012 하락 |
| 40 | 0.988 | 22,260 | 0.409 | △409배 비용 증가, MRR 변화 없음 |
| 50 | 0.988 | 22,904 | 0.496 | △496배 비용 증가, MRR 변화 없음 |

이 결과는 **Top-K = 5가 비용 대비 성능의 최적점(Sweet Spot)**임을 실증하며, 5장에서 설명한 "Lost in the Middle" 현상 방어를 위한 Top-K 제한 설계의 타당성을 수치적으로 뒷받침한다.

**발견 3: Dirty Tree 상태에서의 성능 이탈 (bench-6392ad6a)**

동일한 Top-K = 40 조건에서 MRR이 0.988 → 0.940으로 하락한 유일한 사례다. 이 실행은 Git 커밋 없이 로컬에서 임시로 하이브리드 검색 가중치(Dense/Sparse Pool 비중 또는 RRF K 상수)를 변경한 상태에서 수행된 것으로 추정된다. 이는 **하이브리드 검색의 하이퍼파라미터가 최적값을 벗어났을 때의 성능 페널티**를 시사하며, 설정 변경 시 반드시 Git 커밋을 동반해야 하는 실험 거버넌스의 중요성을 입증한다.

## 6.3 정성 분석: 사례 기반 성능 포렌식 (Case Study 10선)

### 분석 방법론

80문항 평가셋 중 **가장 완벽한 성공 사례 5건**과 **가장 처참한 실패 사례 5건**을 선별하여, 각 사례에 대해 [질문 → 검색된 컨텍스트 → LLM 답변 → 판정 근거]의 4단계 포렌식을 수행하였다. 성공 사례는 시스템의 강점을 실증하고, 실패 사례는 향후 개선 방향의 근거를 제공한다.

### ★ 성공 사례 1: 팩토이드형 예산 추출 — 완벽한 단발 질의 응답

**질문 ID**: Q001

**질문 원문**: (사)벤처기업협회의 "2024년 벤처확인종합관리시스템 기능 고도화 용역사업"에 배정된 총 소요예산은 부가가치세를 포함하여 얼마입니까?

**검색된 컨텍스트 핵심 발췌**:

검색된 7개 청크 중 [1]번 청크에 다음 내용이 포함되어 있었다:
> "□ (소요예산) 352,000,000원(부가가치세 포함)"

메타데이터 필터가 정확히 "(사)벤처기업협회"로 작동하여, 해당 기관의 문서 청크만 검색 범위에 포함되었다. 7개 청크 모두 사업 금액 = 352,000,000원으로 일관된 메타데이터를 보유하고 있어, 크로스 체크가 자연스럽게 이루어졌다.

**LLM 답변 전문**:
> 핵심 답변: 총 소요예산은 352,000,000원(부가가치세 포함)입니다 [1].
> 근거: 사업개요에 "소요예산 352,000,000원(부가가치세 포함)"으로 명시되어 있습니다 [1].
> 문서에 없는 항목: 없음

**판정 결과**: F:1.00 / AR:1.00 / CP:1.00 / CR:1.00 / AC:1.00 — **전 지표 만점**

**성공 요인 분석**: 이 질문은 단일 팩토이드(사실 확인형)로, 정답이 문서 내 한 곳에 명확히 기술되어 있다. 메타데이터 필터(기관명)가 정확히 작동하여 검색 공간이 해당 기관 문서로 좁혀졌고, [1]번 청크에 정답이 온전히 포함되어 있었다. LLM은 컨텍스트 근거를 명시([1])하면서 환각 없이 정확한 답변을 생성하였다.

### ★ 성공 사례 2: 복합 일정 추출 — 2개 시점의 동시 정확 추출

**질문 ID**: Q002

**질문 원문**: (사)벤처기업협회의 "2024년 벤처확인종합관리시스템 기능 고도화" 사업에서 시스템의 1차 오픈과 2차 그랜드 오픈(Grand Open)은 각각 언제로 예정되어 있나요?

**검색된 컨텍스트 핵심 발췌**:

[1]번 청크에 다음 내용이 포함:
> "* 1차 오픈 : 2024.07.10. 성과조건부 및 스톡옵션 오픈"
> "** 2차 오픈 : 2024.09월 中 GrandOpen"

**LLM 답변 전문**:
> 핵심 답변:
> - 1차 오픈: 2024.07.10. (성과조건부 및 스톡옵션 오픈) [1]
> - 2차 그랜드 오픈: 2024년 9월 중 (정확한 일자는 문서에 명시되어 있지 않음) [1]
> 문서에 없는 항목: 2차 그랜드 오픈의 정확한 일자(연·월·일 수준의 특정 날짜)는 제공된 문서에서 해당 정보를 찾을 수 없습니다 [1].

**판정 결과**: F:1.00 / AR:1.00 / CP:1.00 / CR:1.00 / AC:1.00 — **전 지표 만점**

**성공 요인 분석**: 이 사례의 핵심 가치는 **"모르는 것을 모른다고 답한 정직성"**에 있다. LLM은 1차 오픈의 정확한 날짜(2024.07.10)를 명시하면서도, 2차 오픈에 대해서는 "2024년 9월 중"이라는 원문의 불확정 표현을 그대로 인용하고, "정확한 일자는 문서에 명시되어 있지 않음"이라고 솔직하게 고지하였다. 이것이 Faithfulness 1.00의 핵심이다 — 환각을 생성하지 않은 것.

### ★ 성공 사례 3: 네거티브 질문 — 문서에 없는 정보의 정확한 부재 판별

**질문 ID**: Q004

**질문 원문**: "(사)부산국제영화제 2024년 온라인서비스 재개발 및 행사지원시스템 공급" 사업을 전담할 주관사업자 선정과 관련하여, 제안서 기술평가위원들의 개인 연락처 및 기피 소속 부서 명단은 제안요청서의 어느 항목에서 확인할 수 있습니까?

**LLM 답변 핵심**:
> 결론: 제안요청서에서 기술평가위원들의 개인 연락처와 기피 소속 부서 명단은 확인할 수 없습니다. 제공된 문서에서 해당 정보를 찾을 수 없습니다 [1][6][3].

**판정 결과**: F:1.00 / AR:1.00 / CP:1.00 / CR:1.00 / AC:1.00 — **전 지표 만점**

**성공 요인 분석**: 이 질문은 **함정 질문(Trap Question)**이다. 평가위원의 개인 연락처는 어떤 공공입찰 RFP에서도 공개하지 않는 정보다. 미숙한 RAG 시스템은 "별지서식"이나 "평가위원회" 관련 청크를 근거로 무리하게 답변을 구성하려 할 수 있다. 그러나 BidMate RAG는 **"문서에서 해당 정보를 찾을 수 없습니다"**라고 정확히 판별하였다. 이는 Constraints 프롬프트의 "컨텍스트에 없으면 없다고 답하라" 지시가 정확히 작동한 결과다.

### ★ 성공 사례 4: 오타 내성 — 의도적으로 훼손된 질문에 대한 강건한 응답

**질문 ID**: Q005

**질문 원문**: (사)항쿡대학슈포츠혀비회에셔 추찐하는 "KUSF 체특쟈 기룩 관뤼시스탬 기발" 사압의 쳬종 걔약 만료일(종료일)은 언재까쥐 입니가?

**LLM 답변 핵심**:
> 핵심 답변: 최종 계약 만료일(종료일)은 "계약체결 후 2024년 12월 31일까지"로, 즉 2024년 12월 31일입니다 [1].

**판정 결과**: F:1.00 / AR:1.00 / CP:1.00 / CR:1.00 / AC:1.00 — **전 지표 만점**

**성공 요인 분석**: 질문의 거의 모든 글자가 의도적으로 오타 처리되었음에도(`항쿡대학슈포츠혀비회` → 한국대학스포츠협의회, `체특쟈 기룩 관뤼시스탬` → 체육특기자 경기기록 관리시스템), 시스템은 정확한 문서를 검색하고 올바른 답변을 생성하였다. 이는 **Dense 임베딩의 의미론적 유사도 기반 검색**이 표층적 오타에 강건함을 입증한다. BM25(Sparse) 검색만으로는 이 질문의 키워드가 원문과 전혀 매칭되지 않아 검색 자체가 불가능했을 것이다.

### ★ 성공 사례 5: 단순 예산 추출 — 다중 근거 교차 검증

**질문 원문**: (재)예술경영지원센터에서 발주한 "통합 정보시스템 구축 사전 컨설팅" 사업의 총 예산은 얼마로 책정되어 있습니까?

**LLM 답변 핵심**:
> 핵심 답변: 총 예산은 50,000,000원(금 오천만원, VAT 포함)입니다 [1].
> 근거: 제안요청서의 사업일반 항목에 "사업예산 : 50,000,000(금 오천만원/VAT포함)"로 명시되어 있습니다 [1]. 문서 메타데이터(헤더)에도 사업 금액이 50,000,000원으로 반복 표기되어 있습니다 [2][3][4][5][6][7].

**판정 결과**: F:1.00 / AR:1.00 / CP:1.00 / CR:1.00 / AC:1.00 — **전 지표 만점**

**성공 요인 분석**: LLM은 본문 [1]에서 정답을 추출한 뒤, 나머지 6개 청크([2]~[7])의 메타데이터에서 동일 금액이 반복 표기됨을 자발적으로 교차 확인하여 보고하였다. 이러한 **다중 출처 교차 검증(Cross-reference)**은 답변 신뢰도를 극대화하는 행동으로, Constraints 프롬프트의 "출처를 반드시 명시하라" 지시가 효과적으로 작동한 결과다.

# 7. 기술 의사결정 및 트러블슈팅

본 장은 프로젝트 개발 과정에서 마주한 핵심 기술적 난관과, 이를 해결하기 위해 내린 설계 의사결정을 영역별로 서술한다. 각 섹션은 **"어떤 문제가 있었고, 어떤 설계를 선택했으며, 그 결과 성능이 어떻게 변화했는가"**의 구조로 정리하였다.

---

## 7.1 검색 파이프라인 고도화

### 7.1.1 문제 인식: Dense 단일 검색의 한계

프로젝트 초기 파이프라인은 Dense(임베딩) 단일 검색만을 사용하였다. 최초 통합 벤치마크에서 MRR = 0.000이 기록된 후 검색 결과를 수동 분석한 결과, **정답이 포함된 청크가 검색 후보에는 존재하였으나 순위가 낮아 LLM이 활용하지 못하는 패턴**이 확인되었다. Dense 검색은 의미론적 유사도 기반으로 후보를 추출하지만, 질문과 컨텍스트 간 정밀한 관련성 판별에는 구조적 한계가 있었다.

### 7.1.2 Cross-Encoder 리랭커 도입



이를 해결하기 위해 **Cross-Encoder 기반 리랭킹 파이프라인**을 도입하였다. Cross-Encoder는 (질문, 청크) 쌍을 동시에 입력받아 관련성 점수를 직접 산출하는 모델로, Dense Bi-Encoder 대비 정밀도가 현저히 높다.

| 구분 | 도입 전 | 도입 후 |
|---|---|---|
| 파이프라인 | Dense Top-K → LLM | Dense Top-K → **Cross-Encoder 재정렬** → 상위 N개 → LLM |
| 정답 청크 순위 | 중·하위권 분산 | 최상위 집중 |
| 추가 Latency | — | +200~500ms (모델 추론) |

### 7.1.3 하이브리드 검색 전환: Dense + Sparse + RRF 융합



프로젝트에서 **가장 큰 아키텍처 전환**이다. Dense(임베딩) 단일 검색에서 **Dense + Sparse(BM25) 이중 경로 병렬 검색 + RRF(Reciprocal Rank Fusion) 스코어 융합**으로 전환하였다.

| 검색 방식 | 강점 | 약점 | 단독 사용 시 실패 유형 |
|---|---|---|---|
| Dense | 의미론적 유사도, 오타 내성 | 키워드 정확 매칭 불가 | "352,000,000원" 숫자 검색 실패 |
| Sparse (BM25) | 키워드 정확 매칭, 숫자/고유명사 | 동의어/오타 처리 불가 | "항쿡대학슈포츠혀비회" 매칭 불가 |
| **하이브리드** | **양쪽 강점 결합** | 하이퍼파라미터 튜닝 필요 | — |

하이브리드 전환 후, Dense만으로 놓치던 **숫자 기반 팩토이드 질문**(예: "예산이 얼마인가?")의 검색 정확도가 현저히 향상되었고, Dense의 오타 내성은 그대로 유지되어 성공 사례 4(Q005)와 같은 훼손된 질문에도 강건한 응답이 가능해졌다.

### 7.1.4 Hard Filter에서 Soft Boost로의 전환



초기 구현에서는 LLM이 "이 질문의 답은 '사업 개요' 섹션에 있을 것"으로 추론하면, ChromaDB의 `where_document` 필터로 해당 섹션의 청크만 검색하는 **Hard Filter** 방식을 적용하였다.

그러나 이 접근법은 **LLM의 섹션 추론이 틀릴 경우 정답 청크가 검색 대상에서 완전히 배제되는** 치명적 결함을 내포하고 있었다. 실제로 "무상 하자보수 기간은?"이라는 질문에 대해 LLM이 "프로젝트 지원" 섹션을 추론했지만, 정답은 "사업 개요" 섹션에 있어 정답 청크가 아예 검색되지 않는 사례가 관찰되었다.

이를 3단계에 걸쳐 **Soft Boost** 방식으로 전환하였다:

1. LLM rewrite 응답을 JSON 구조로 변경하여 `section_hint`를 추출
2. `where_document` Hard Filter를 제거
3. 동적 앵커 기반 Soft Boost로 대체하되, **MAX_TOTAL_BOOST = 0.15** 상한을 설정하여 부스팅이 Cross-Encoder 본래 점수를 뒤집지 못하게 방어

| 구분 | Hard Filter | Soft Boost |
|---|---|---|
| 섹션 힌트가 맞을 때 | ✅ 정답만 검색 | ✅ 정답 순위 상승 |
| 섹션 힌트가 틀릴 때 | ❌ **정답 완전 차단** | ✅ 다른 경로로 정답 도달 가능 |
| 부스팅 상한 | 없음 (전부 또는 전무) | 0.15 (Cross-Encoder 보호) |

## 7.2 평가 체계 수립: Judge v1에서 v2로의 전환

### 7.2.1 문제: MRR = 0.000이 의미하는 것



최초 4회 연속 벤치마크에서 MRR = 0.000이 기록되었다. 이것은 RAG 파이프라인의 검색 실패가 아니라, **LLM Judge v1의 채점 로직이 LLM 응답 포맷과 불일치하여 정답을 오답으로 판정한 평가 엔진 버그**였다.

BidMate RAG의 LLM은 "핵심 답변 / 근거 / 문서에 없는 항목"이라는 구조화된 포맷으로 응답하는데, Judge v1의 단순 문자열 매칭 로직이 이 구조를 파싱하지 못한 것이다.

### 7.2.2 Judge v2 재설계

| 요소 | Judge v1 | Judge v2 |
|---|---|---|
| 채점 방식 | 문자열 부분 매칭 | LLM 기반 의미론적 판정 |
| 세부 지표 | Pass/Fail 이분법 | Faithfulness, Answer Relevance, Context Precision, Context Recall, Accuracy 5개 ratio score |
| 근거 처리 | 무시 | 근거 인용([n])의 정확성까지 평가 |

Judge v2 적용 후 **MRR이 즉시 0.000 → 1.000으로 정상화**되었다. 시스템의 실제 성능이 원래 높았으나, 측정 도구의 버그가 성능을 은폐하고 있었던 것이다.

> **핵심 교훈**: 측정 도구의 정확성이 곧 성능 지표의 정확성이다. 평가 엔진이 잘못되면 아무리 파이프라인을 개선해도 지표에 반영되지 않는 맹점이 발생한다. 따라서 **파이프라인 고도화 이전에 평가 체계의 무결성부터 확보**해야 한다.

## 7.3 멀티턴 대화 엔진 구축

### 7.3.1 아키텍처 분리: 단일턴과 멀티턴의 이원화



초기에는 모든 질의가 동일한 Retriever 경로를 공유했다. 그러나 멀티턴 질문("그렇다면 이러한 법적인 문제를 방지하기 위해…")에서는 이전 대화의 맥락을 반영한 **쿼리 재작성(Query Rewriting)**이 선행되어야 하므로, 검색 흐름 자체가 근본적으로 달라야 한다.

| 구분 | 도입 전 | 도입 후 |
|---|---|---|
| 단일턴 | 공유 Retriever | **RAGRetriever** (독립 최적화) |
| 멀티턴 | 공유 Retriever | **MultiturnRetriever** (재작성 + 슬롯 메모리 + Reranker) |

### 7.3.2 LLM Rewrite 엔진의 구현과 안정화

멀티턴 쿼리 재작성 엔진의 구현과 안정화는 프로젝트에서 **가장 밀도 높은 개발이 집중된 구간**이었다. 구현 → 엣지 케이스 발견 → 수정의 반복이 연속으로 이루어졌으며, 주요 해결 과제는 다음과 같다.

| 단계 | 해결한 문제 |
|---|---|
| 인터페이스 설계 | rewrite() 추상 인터페이스 정의 및 OpenAI·로컬 LLM 양쪽 구현 |
| 시나리오 A 대응 | 로컬 LLM(HFLocal)에서도 rewrite 가능하도록 generator 재사용 |
| 엣지 케이스 | 로컬 모델이 빈 응답(None)을 반환하는 경우 처리 |
| 리팩터링 | 하드코딩된 재작성 로직을 추상 인터페이스로 통합 |
| **상태 불일치** | **재작성된 쿼리(resolved_query)가 부스팅 로직에 전달되지 않는 버그** |
| **소유권 문제** | **메모리 상태가 여러 모듈에 분산되어 일관성 파괴** → Retriever로 단일화 |
| 추적 가능성 | 디버깅 비활성화 시에도 memory_state 노출 |
| 안전 장치 | 잘못된 설정값(음수 토큰/타임아웃)으로 인한 무한 대기 방지 |

특히 상태 불일치와 소유권 문제는 **단일턴 테스트에서는 절대 드러나지 않는, 멀티턴 고유의 상태 관리 버그**였다. 이후 rewrite 타임아웃 시 원본 쿼리로 자동 fallback하는 안전 장치가 추가되어 멀티턴 엔진이 최종 안정화되었다.

| 구분 | 도입 전 | 도입 후 |
|---|---|---|
| "그렇다면 이것은?" | 원본 질문 그대로 검색 → 맥락 없는 결과 | LLM rewrite로 대명사 치환 → resolved_query로 검색 |
| 실패 패턴 | 6.3절 실패 사례 1(Q049) | 맥락 반영된 정확한 검색 |

> **핵심 교훈**: 멀티턴 대화는 단일턴과 비교할 수 없는 복잡도를 가진다. 단일턴에서는 존재하지 않는 상태 관리(메모리 소유권, resolved_query 일관성, 타임아웃 fallback) 문제가 멀티턴에서만 폭발적으로 발현된다.

## 7.4 프롬프트 엔지니어링 체계화

### 7.4.1 단일 프롬프트에서 5-Tier 조합형 아키텍처로의 전환



초기 시스템은 단일 시스템 메시지를 하드코딩하여 LLM에 전달하였다. 이 방식은 프롬프트 변경 시 코드 수정이 필요하고, A/B 테스트가 불가능하며, 시나리오 A(로컬 LLM)와 시나리오 B(상용 API)에 동일한 프롬프트를 강제하는 문제가 있었다.

이를 해결하기 위해 **5-Tier 프롬프트 아키텍처**(Persona × Constraints × Format × Few-shot × CoT)를 설계하고, 각 Tier를 독립된 YAML 설정으로 분리하였다.

| 구분 | 도입 전 | 도입 후 |
|---|---|---|
| 프롬프트 정의 | 코드 내 하드코딩 | YAML 1장 = 실험 1건 |
| 변경 시 | 코드 수정 + 재배포 | YAML 수정만으로 즉시 반영 |
| 조합 실험 | 수동, 비체계적 | **5,400개 조합 자동 생성** 후 배치 실행 |
| 다국어 대응 | 미고려 | 5,400개 YAML에 "한국어 응답 지침" 일괄 주입 |

### 7.4.2 Citation 사전 주입 시스템



LLM이 답변에서 [1], [2] 등의 출처 번호를 인용할 때, 해당 번호가 실제 검색 결과의 순서와 불일치하는 문제가 관찰되었다. 이를 해결하기 위해 **LLM에 컨텍스트를 전달할 때 각 청크에 [1], [2] 번호를 사전 주입(pre-injection)하여**, 답변의 출처 번호와 근거 카드가 정확히 1:1로 매칭되도록 보장하였다.

## 7.5 데이터 품질 및 인덱싱 안정화

### 7.5.1 Document ID 매칭 오류 해결

인덱싱 초기에 다음과 같은 데이터 품질 문제가 연쇄적으로 발생하였다:

| 문제 | 증상 | 해결 방법 |
|---|---|---|
| NaN doc_id 저장 | 일부 청크의 문서 소속이 누락되어 검색 필터 우회 | notna() 명시 체크 로직 추가 |
| 공고 번호 불일치 | 문서-청크 간 ID 매칭 실패로 엉뚱한 문서의 청크 반환 | 공고 번호 기반 document ID 통일 |
| 파일명 fallback 부재 | 공고 번호가 없는 문서에서 매칭 완전 실패 | 파일명 기반 fallback 매칭 도입 |
| Cleaner/Validator 불안정 | 전처리 후 데이터 스키마 불일치 | 스키마 검증 및 필터 안정화 |

이 문제들은 개별적으로는 사소해 보이지만, **검색 필터가 엉뚱한 기관의 문서를 반환하는 치명적 오류**로 이어졌다. 특히 실패 사례 1(Q049)과 같이, 질문에서 언급되지 않은 기관(고양도시관리공사)의 문서가 검색되는 현상의 근본 원인 중 하나였다.

## 7.6 비용 추적 및 최적화 (Cost Governance)

### 토큰 단위 비용 자동 추적

본 시스템은 모든 LLM 호출(생성, 재작성, 평가)에 대해 prompt_tokens, completion_tokens, total_tokens를 자동으로 기록하며, `tracking/pricing.py`의 `calc_llm_cost` 함수가 모델별 단가표를 참조하여 실시간 비용을 산출한다.

| 비용 발생 지점 | 모델 | 평균 비용/건 |
|---|---|---|
| 답변 생성 | gpt-5-mini | $0.003~$0.010 |
| 멀티턴 재작성 | gpt-5-mini | $0.001~$0.003 |
| LLM Judge 평가 | gpt-4o-mini | $0.002~$0.005 |
| 임베딩 (인덱싱) | text-embedding-3-small | $0.15 (1회성, 전체) |

### Top-K와 비용의 상관관계

6.1절의 실험 로그에서 관찰된 바와 같이, Top-K = 5에서의 건당 비용은 $0.001인 반면, Top-K = 50에서는 $0.496으로 **496배 증가**하였다. 이는 LLM에 전달되는 컨텍스트의 토큰 수가 Top-K에 비례하여 증가하기 때문이다. Top-K를 10배 늘려도 MRR 개선이 없다는 실험 결과와 결합하면, **Top-K = 5가 비용 대비 성능의 절대적 최적점**임이 재확인된다.

## 8.2 시나리오 A/B 비교를 통한 엔진 선택 가이드라인

### 의사결정 프레임워크

조직이 BidMate RAG를 프로덕션 배포할 때, 시나리오 A(로컬 LLM)와 시나리오 B(상용 API) 중 어떤 엔진을 선택해야 하는지는 다음 3가지 축으로 판단한다.

#### 축 1: 데이터 보안 요건

| 보안 수준 | 권장 시나리오 | 근거 |
|---|---|---|
| **내부 기밀 등급** (국가 R&D, 방산, 금융) | 시나리오 A (로컬) | 문서 내용이 외부 서버로 전송되면 안 됨 |
| **일반 공개 등급** (공개 입찰 RFP) | 시나리오 B (API) 허용 | RFP 자체가 공개 문서이므로 보안 리스크 낮음 |
| **하이브리드** | 검색은 로컬, 생성만 API | 청크 텍스트만 API로 전송, 전체 문서 전송 방지 |

#### 축 2: 품질 요구 수준

| 품질 기대치 | 권장 시나리오 | 근거 |
|---|---|---|
| **Faithfulness ≥ 0.95 필수** | 시나리오 B (GPT-5) | 4-bit 양자화 로컬 모델은 복잡한 논리 추론에서 성능 저하 가능 |
| **Faithfulness ≥ 0.80 허용** | 시나리오 A 검토 가능 | 파인튜닝으로 도메인 특화 시 달성 여지 있음 |
| **응답 일관성 중시** | 시나리오 B | 상용 API의 temperature=0 설정으로 재현 가능한 답변 보장 |

#### 축 3: 비용 구조 선호

| 비용 구조 | 권장 시나리오 | 근거 |
|---|---|---|
| **초기 투자 최소화** | 시나리오 B (API) | GPU 서버 없이 즉시 시작, 종량제 과금 |
| **장기 운영 비용 최소화** | 시나리오 A (로컬) | GPU 서버 고정비만 발생, 건당 추가 비용 $0 |
| **BEP (손익분기점)** | 아래 분석 참조 | — |

### 비용 효율성 분석: BEP (Break-Even Point) 추정

시나리오 A와 B의 손익분기점을 추산하여, 어떤 사용량에서 로컬 GPU 투자가 정당화되는지 분석한다.

**시나리오 B 비용 구조 (변동비)**:

| 비용 항목 | 단가 | 비고 |
|---|---|---|
| 답변 생성 | $0.005/건 | GPT-5-mini, Top-K=5 기준 |
| 멀티턴 재작성 | $0.002/건 | 재작성 발생 시에만 |
| 평가 (Judge) | $0.003/건 | 벤치마크 시에만 |
| **운영 기준 건당 비용** | **~$0.007/건** | 재작성 50% 발생 가정 |

**시나리오 A 비용 구조 (고정비)**:

| 비용 항목 | 월 비용 | 비고 |
|---|---|---|
| GCP L4 GPU 인스턴스 | ~$500/월 | g2-standard-8 기준 |
| 스토리지 (SSD) | ~$50/월 | 500GB 기준 |
| **월 고정비 합계** | **~$550/월** | API 비용 $0 |

**BEP 계산**:

> 월 질의 수 × $0.007 = $550
> **BEP = 약 78,571건/월 (약 2,619건/일)**

즉, **일일 평균 2,619건 이상의 질의가 발생하는 환경**에서는 시나리오 A(로컬 GPU)가 시나리오 B(API)보다 경제적이다. 반대로, 일일 수백 건 수준의 사내 도구 용도라면 시나리오 B가 초기 투자 없이 즉시 운영 가능하므로 더 적합하다.

### 권장 배포 전략 매트릭스

| 조직 유형 | 일일 질의량 | 보안 등급 | 권장 시나리오 |
|---|---|---|---|
| 소규모 컨설팅 팀 (5~10명) | 50~200건 | 일반 | **시나리오 B** |
| 중규모 입찰 전문 조직 (20~50명) | 500~2,000건 | 일반 | **시나리오 B** (BEP 미달) |
| 대규모 공기업/금융기관 | 3,000건+ | 내부 기밀 | **시나리오 A** |
| 방산/연구기관 | 무관 | 보안 필수 | **시나리오 A** (비용 불문) |

## 8.3 현재 한계 및 향후 고도화 로드맵

### 식별된 한계점

6.3절의 실패 사례 분석과 실험 로그 인과 분석을 종합하여, 현 시스템의 한계를 다음과 같이 정리한다.

| 한계 영역 | 구체적 증상 | 영향도 | 근본 원인 |
|---|---|---|---|
| **멀티턴 맥락 복원** | "그렇다면", "이러한" 등 대명사 참조 실패 (Q049) | 높음 | 슬롯 메모리의 주제 추적 정밀도 부족 |
| **심층 검색** | 성능 목표 수치 등 세부 스펙이 Top-K 밖에 위치 (Q061) | 높음 | dense 유사도만으로는 수치 정보의 중요도 판별 불가 |
| **컨텍스트 내 정답 인식** | 정답이 청크 내에 있으나 LLM이 간과 (Q056) | 중간 | Lost in the Middle + 용어 미세 불일치 |
| **평가셋 품질** | 문서에 없는 정보를 묻는 질문 존재 (Q050, Q071) | 낮음 | 평가셋 작성 시 정답 존재 여부 사전 검증 미흡 |

### 향후 고도화 로드맵

**Phase 1: 검색 정밀도 강화 (단기)**

- Sparse 검색에서 숫자/단위 포함 청크에 대한 가중치 부스팅 도입
- `where_document` 앵커를 활용한 본문 내 키워드 기반 필터 강화
- 멀티턴 슬롯 메모리의 "주제어" 추적 정밀도 개선

**Phase 2: 로컬 LLM 품질 향상 (중기)**

- RFP 도메인 특화 PEFT 파인튜닝 (LoRA/QLoRA)
- Gemma-4-E4B-it 및 Qwen2.5-7B 각각에 대해 500건 이상의 도메인 데이터로 파인튜닝
- 시나리오 A의 Faithfulness를 0.90 이상으로 끌어올릴 경우, 보안이 필요한 기관에서의 실전 배포 가능

**Phase 3: 프로덕션 안정화 (장기)**

- Next.js 사용자 UI와 FastAPI 백엔드의 프로덕션 배포
- 모니터링 대시보드(Latency P95, Error Rate, Cost/day) 구축
- A/B 테스트 프레임워크: 동일 질의에 대해 시나리오 A/B 답변을 병렬 생성하고 사용자 선호도 피드백 수집

## 8.4 결론

BidMate RAG 프로젝트는 공공입찰 RFP 101건을 대상으로, **데이터 무결성 확보 → 구조 보존형 전처리 → 하이브리드 검색 오케스트레이션 → Cross-Encoder 재정렬 → 5-Tier 프롬프트 기반 생성**이라는 엔드투엔드 파이프라인을 구축하여, 시나리오 B에서 **MRR 0.988, 건당 비용 $0.001 (Top-K=5 기준)**이라는 정량적 성과를 달성하였다.

사례 분석에서 확인된 바와 같이, 성공 사례의 100%가 **메타데이터 필터의 정확한 작동 + 환각 방지 Constraints의 준수**라는 두 축 위에서 달성되었다. 반면, 실패 사례의 100%가 **검색 단계(Retrieval)의 한계**에서 기인하였으며, LLM의 환각으로 인한 실패는 단 한 건도 관찰되지 않았다.

이는 RAG 시스템의 성능 향상 전략이 **"더 큰 LLM"이 아니라 "더 정밀한 검색"**에 집중되어야 함을 시사한다. BidMate RAG 시스템은 이 원칙에 충실하여, 7단계 검색 오케스트레이션과 4단계 Reranker 파이프라인이라는 업계 최고 수준의 검색 정밀도 아키텍처를 구현하였다.

시나리오 A(로컬 LLM)와 시나리오 B(상용 API)의 이원화 전략은, 보안 요건과 비용 구조가 극단에 있는 조직 모두에게 적합한 솔루션을 제공한다. 특히 BEP 분석(일일 2,619건)을 통해, 각 조직의 상황에 맞는 합리적 엔진 선택이 가능함을 정량적으로 입증하였다.

**BidMate RAG는 "질문에 대해 문서 근거를 명시하며 정확하게 답변하는" 지식 근거주의(Evidence-grounded) 체계로서, 공공입찰 컨설팅의 디지털 전환을 위한 견고한 기술적 토대를 확립하였다.**
