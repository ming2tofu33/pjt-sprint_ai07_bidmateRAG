# 4. RAG 시스템 아키텍처 (Scenario Design)

## 4.1 시나리오 A: 로컬 거대 언어 모델 최적화 — Native FP16 모델 원천 지능 보존 전략

### 설계 철학: 왜 로컬 LLM인가

공공입찰 RFP 문서에는 사업 예산, 기관 내부 구조, 입찰 자격 조건 등 **민감한 행정 정보**가 다수 포함되어 있다. 기업 보안 정책상 이러한 문서 내용이 외부 API(OpenAI, Google 등)의 서버로 전송되는 것을 허용하지 않는 경우가 빈번하다. 시나리오 A는 이러한 **프라이빗 환경(Air-gapped Environment)**을 전제로, 외부 네트워크 접근 없이 조직 내부의 GPU 서버에서 모든 추론을 완결하는 아키텍처를 설계하였다.

그러나 로컬 LLM 운영에는 다음과 같은 근본적인 엔지니어링 제약이 존재한다.

| 제약 요소 | 구체적 문제 | 영향 |
|---|---|---|
| **VRAM 한계** | NVIDIA L4 GPU의 24GB VRAM으로는 7B 파라미터 모델의 FP32 로딩 불가 (약 28GB 필요) | 양자화(Quantization) 필수 |
| **추론 속도** | 로컬 GPU에서의 토큰 생성 속도는 클라우드 API 대비 2~5배 느림 | 사용자 경험 저하 |
| **한국어 성능** | 대부분의 오픈소스 LLM은 영문 중심 학습, 한국어 도메인 특화 부족 | 답변 품질 하락 |
| **프롬프트 길이** | 컨텍스트 윈도우가 제한적 (4K~8K 토큰) | 검색된 청크의 투입량 제한 |

### 대상 모델 선정: Gemma-4-E4B-it 및 Qwen2.5-7B

시나리오 A에서는 두 가지 로컬 오픈소스 LLM을 대상으로 비교 실험을 수행하였다.

#### 모델 비교 매트릭스

| 속성 | Gemma-4-E4B-it | Qwen2.5-7B-Instruct |
|---|---|---|
| **개발사** | Google DeepMind | Alibaba Cloud |
| **파라미터 규모** | 4B 클래스 | 7B |
| **학습 데이터 언어** | 다국어 (한국어 포함) | 다국어 (한국어 최적화) |
| **라이선스** | Gemma License | Apache 2.0 |
| **추론 프레임워크** | vLLM / Ollama | HuggingFace Transformers |
| **프로바이더 설정** | `configs/providers/scenario_a/` | `configs/providers/scenario_a/llms/qwen2.5-7b.yaml` |
| **주요 강점** | 경량화, 빠른 추론 | 한국어 도메인 이해도 |

#### Qwen2.5-7B 선정 근거

Qwen2.5-7B-Instruct는 Alibaba Cloud의 Qwen 시리즈 중 한국어 최적화 모델로 평가받고 있다. 시스템의 `configs/providers/scenario_a/llms/qwen2.5-7b.yaml` 설정 파일에 다음과 같이 등록되어 있다.

| 설정 키 | 값 |
|---|---|
| `provider` | huggingface |
| `model` | Qwen/Qwen2.5-7B-Instruct |
| `description` | Qwen2.5 7B 한국어 최적화 모델 |

이 모델은 HuggingFace 생태계를 통해 로딩되며, 시나리오 A의 파이프라인에서 `build_llm_provider` 함수가 프로바이더 설정을 읽어 자동으로 인스턴스를 생성한다.

### VRAM 다이어트 전략: 양자화와 추론 최적화

24GB VRAM이라는 물리적 제약 아래에서 7B 파라미터 모델을 구동하기 위해, 본 시스템은 다음과 같은 다단계 최적화 전략을 적용하였다.

**1단계: 4-bit Quantization (bitsandbytes)**

모델의 가중치를 FP32(32비트) 또는 FP16(16비트)에서 INT4(4비트)로 양자화한다. 이를 통해 VRAM 사용량이 약 1/4~1/8로 축소된다.

| 정밀도 | 7B 모델 예상 VRAM | L4 GPU (24GB) 적합성 |
|---|---|---|
| FP32 | ~28GB | ❌ 초과 |
| FP16 | ~14GB | ⚠️ 빠듯 |
| INT8 | ~7GB | ✅ 여유 |
| INT4 | ~3.5GB | ✅ 충분 |

단, 양자화는 필연적으로 **모델의 원천 지능(Native Intelligence)을 훼손**한다. 특히 4-bit 양자화 시 한국어 조사 처리, 복합 논리 추론, 숫자 정확성에서 성능 저하가 관찰될 수 있다. 이것이 시나리오 A의 본질적 트레이드오프(Trade-off)다.

**2단계: vLLM / Ollama 기반 추론 서버 분리**

모델 로딩과 추론 요청을 분리하여, 한 번 로딩된 모델이 메모리에 상주하면서 다수의 질의를 효율적으로 처리하도록 구성한다. 이를 통해 매 질의마다 모델을 재로딩하는 비효율을 제거하였다.

**3단계: PEFT(Parameter-Efficient Fine-Tuning) 준비**

`pyproject.toml`의 `ml` 의존성 그룹에 `peft`, `trl`, `datasets` 등의 파인튜닝 라이브러리가 포함되어 있으며, `scripts/train_peft.py` 스크립트가 이미 구현되어 있다. 이를 통해 향후 RFP 도메인 특화 파인튜닝을 수행할 수 있는 기반을 마련하였다.

### 시나리오 A의 파이프라인 흐름

시나리오 A에서의 전체 데이터 흐름은 다음과 같다.

| 단계 | 처리 내용 | 실행 환경 |
|---|---|---|
| ① 질문 입력 | 사용자 질문 수신 | 로컬 서버 |
| ② 임베딩 | BAAI/bge-m3 임베딩 벡터 생성 | 로컬 GPU |
| ③ 검색 | ChromaDB에서 Top-K 청크 검색 | 로컬 스토리지 |
| ④ 컨텍스트 조립 | 검색 결과 + 시스템 프롬프트 결합 | CPU |
| ⑤ 추론 | Gemma-4-E4B-it 또는 Qwen2.5-7B로 답변 생성 | 로컬 GPU |
| ⑥ 응답 반환 | 생성된 답변 출력 | 로컬 서버 |

**핵심 설계 원칙: 전 과정에서 단 한 바이트도 외부 서버로 유출되지 않는다.**

## 4.2 시나리오 B: 고성능 상용 API 파이프라인 — 품질 천장 확인용 레퍼런스 아키텍처

### 설계 철학: 성능의 절대 상한선은 어디인가

시나리오 B는 시나리오 A와 정반대의 설계 철학을 따른다. **비용과 프라이버시를 양보하는 대신, 현존하는 최고 수준의 LLM 성능을 내 파이프라인에 투입하여 RAG 시스템이 도달할 수 있는 품질 천장(Ceiling)을 확인**하는 것이 목적이다.

이 레퍼런스 모델과 시나리오 A 로컬 모델의 성능 간극(Performance Gap)을 정량적으로 측정함으로써, 다음과 같은 전략적 의사결정에 활용한다.

| 의사결정 항목 | 판단 기준 |
|---|---|
| 로컬 모델 품질이 충분한가? | 시나리오 B 대비 Faithfulness/Relevance 차이가 10% 이내인가? |
| 파인튜닝에 투자할 가치가 있는가? | 성능 간극이 파인튜닝으로 좁힐 수 있는 범위인가? |
| 프로덕션 배포 모델 선택 | 비용 대비 품질(Cost-Quality Ratio) 최적점은 어디인가? |

### 시나리오 B 모델 구성

| 속성 | 설정값 |
|---|---|
| **Generation LLM** | GPT-5-mini / GPT-5 / GPT-5-nano |
| **임베딩 모델** | BAAI/bge-m3 (시나리오 간 통일) |
| **Judge 모델** | gpt-4o-mini |
| **API 엔드포인트** | OpenAI API (cloud) |
| **과금 방식** | 토큰 단위 종량제 |

시나리오 B의 핵심은 **임베딩 모델을 시나리오 A와 동일하게 고정**함으로써, 검색(Retrieval) 단계의 변수를 제거하고 **순수하게 생성(Generation) 단계의 LLM 성능 차이만을 분리 측정**할 수 있도록 설계한 것이다. 실험 설정에서 `mode: generation_only`로 지정하면 동일 인덱스(ChromaDB 컬렉션)를 공유하며 LLM만 교체하여 비교할 수 있다.

### 두 시나리오의 전략적 대조

| 비교 축 | 시나리오 A (로컬 LLM) | 시나리오 B (상용 API) |
|---|---|---|
| **데이터 보안** | ✅ 완전 내부 처리 | ⚠️ 외부 API 전송 |
| **응답 품질** | 양자화로 인한 성능 저하 우려 | 최고 수준의 추론 능력 |
| **Latency** | GPU 성능에 종속 (5~30초) | 네트워크 + API 처리 (2~8초) |
| **운영 비용** | GPU 서버 유지 비용 (고정비) | 토큰당 과금 (변동비) |
| **확장성** | GPU 추가 필요 | API 호출 수 증가로 즉시 확장 |
| **한국어 특화** | 모델 선택에 따라 가변 | GPT-5 시리즈의 다국어 지원 |
| **파인튜닝** | PEFT 적용 가능 | 불가 (API 모델) |

## 4.3 초정밀 벡터 데이터베이스(Vector DB) 메타 필터 레이어 구축

### ChromaDB를 선택한 엔지니어링 근거

벡터 데이터베이스의 선택은 RAG 시스템의 검색 성능과 운영 효율성에 직접적인 영향을 미친다. 본 시스템은 다음과 같은 이유로 **ChromaDB**를 채택하였다.

| 선정 기준 | ChromaDB의 강점 |
|---|---|
| **로컬 퍼시스턴스** | `PersistentClient` 지원으로 GPU 서버 로컬 디스크에 직접 저장 |
| **메타데이터 필터링** | `where` 절을 통한 구조화된 필터 native 지원 |
| **Python 생태계** | LangChain, HuggingFace와의 원활한 통합 |
| **경량성** | 별도 서버 프로세스 없이 인프로세스(in-process) 실행 |
| **코사인 유사도** | 기본 거리 함수로 코사인 유사도 지원 |

### 벡터 인덱스 구축 프로세스

3장에서 생성된 14,035개의 청크는 다음과 같은 프로세스를 거쳐 벡터 DB에 인덱싱된다.

| 단계 | 처리 내용 | 수치 |
|---|---|---|
| ① 최소 크기 필터링 | 50자 미만 청크 제외 | 14,035 → 13,951개 |
| ② 임베딩 벡터 생성 | `text_with_meta` 필드를 BAAI/bge-m3로 인코딩 | 1,536차원 벡터 |
| ③ 메타데이터 동시 저장 | 12개 메타데이터 필드를 벡터와 함께 저장 | — |
| ④ 퍼시스턴스 | 로컬 디스크에 영구 저장 | 384MB |

임베딩 비용: 약 $0.15 (207원) — text-embedding-3-small 기준

### 컬렉션 격리 전략: 실험 간 데이터 오염 방지

본 시스템은 동시에 수십 개의 실험을 병렬로 수행할 수 있도록, **실험 모드에 따른 컬렉션 격리 전략**을 구현하였다.

| 실험 모드 | 컬렉션 관리 | 이유 |
|---|---|---|
| `full_rag` | `{실험명}-{프로바이더}-{모델}` 형태로 격리 | 청킹 파라미터가 다르면 인덱스가 달라야 함 |
| `generation_only` | 공용 컬렉션 공유 | 동일 인덱스에서 LLM만 교체하여 비교 |
| `ad-hoc` | `bidmate-{프로바이더}-{모델}` 기본 패턴 | 일회성 질의용 |

이 격리 전략의 핵심은 **청킹 파라미터가 변경되면 반드시 새로운 컬렉션을 생성하여 Stale Chunk(오래된 청크 잔존) 문제를 차단**하는 것이다. `chunk_size=500`으로 생성된 인덱스와 `chunk_size=1000`으로 생성된 인덱스가 혼합되면, 검색 결과의 일관성이 파괴된다.

### Where Clause: 메타데이터 기반 하드 필터

ChromaDB의 `where` 필터는 벡터 유사도 검색 이전에 **물리적으로 검색 범위를 축소**하는 하드 필터(Hard Filter)다. 이는 검색 정밀도와 속도 모두에 극적인 영향을 미친다.

| 필터 유형 | 사용 예시 | 효과 |
|---|---|---|
| **기관 필터** | `{"발주 기관": "국민연금공단"}` | 13,951개 → 해당 기관 문서의 청크만 검색 |
| **도메인 필터** | `{"사업도메인": "교육/학습"}` | 교육 관련 사업의 청크만 검색 |
| **기관유형 필터** | `{"기관유형": "대학교"}` | 대학교 발주 사업만 검색 |
| **범위 필터** | `{"사업 금액": {"$gte": 100000000}}` | 1억 이상 사업만 검색 |
| **연도 필터** | `{"공개연도": 2024}` | 2024년 공개 사업만 검색 |
| **복합 필터** | `{"발주 기관": {"$in": ["A기관", "B기관"]}}` | A 또는 B 기관 문서만 검색 |

이 하드 필터가 없다면, "국민연금공단의 보안 요구사항은?"이라는 질문에 대해 시스템은 13,951개 전체 청크를 코사인 유사도로 비교해야 하며, 보안 관련 키워드가 유사한 타 기관의 문서가 상위권에 침입할 수 있다. 하드 필터 적용으로 검색 대상이 해당 기관의 수백 개 청크로 축소되면, **검색 정밀도와 속도가 동시에 향상**된다.

# 5. 검색 및 생성 고도화 전략

## 5.1 한국어 도메인 특화 임베딩(BAAI/bge-m3) 모델 성능 평가

### 임베딩 모델 선정의 전략적 중요성

RAG 시스템에서 임베딩 모델은 **질문과 문서 청크를 동일한 벡터 공간에 매핑하는 번역기** 역할을 한다. 이 번역기의 품질이 낮으면, 아무리 정교한 검색 알고리즘이나 재정렬 전략을 적용해도 근본적으로 정답 청크를 찾을 수 없다.

공공입찰 RFP 도메인은 일반적인 한국어 텍스트와 현저히 다른 어휘적 특성을 갖는다.

| 특성 | 일반 한국어 텍스트 | RFP 도메인 텍스트 |
|---|---|---|
| 주요 어휘 | 일상 대화, 뉴스 기사 | "지체상금률", "수의계약", "기술성평가" |
| 숫자 중요도 | 보조적 | 핵심적 (예산, 기간, 배점) |
| 고유명사 밀도 | 낮음 | 높음 (기관명, 시스템명, 법률 조항) |
| 표 내부 텍스트 | 드묾 | 매우 빈번 |

### BAAI/bge-m3 채택 근거

본 시스템은 모든 실험에서 임베딩 모델을 **BAAI/bge-m3**로 고정하였다. 이 모델은 실험 설정 파일(`configs/experiments/gemma_fixed_eval.yaml`)에서 다음과 같이 명시되어 있다.

| 설정 키 | 값 |
|---|---|
| `embedding_model` | BAAI/bge-m3 |
| `embedding_provider` | huggingface |

BGE-M3(Multilingual, Multi-Functionality, Multi-Granularity)는 다음과 같은 이유로 선정되었다.

| 선정 기준 | BGE-M3의 강점 |
|---|---|
| **다국어 지원** | 100개 이상 언어 학습, 한국어 포함 |
| **Dense + Sparse 동시 지원** | 하이브리드 검색에 단일 모델로 대응 가능 |
| **벡터 차원** | 1,536차원 — 의미 표현 해상도가 높음 |
| **문맥 윈도우** | 8,192 토큰 — 긴 청크도 온전히 임베딩 가능 |
| **오픈소스** | 로컬 구동 가능 — 시나리오 A와 호환 |

### 임베딩 입력 전략: 메타데이터 접두사 포함 여부

3장에서 설명한 바와 같이, 각 청크는 두 가지 텍스트 필드를 보유한다.

| 필드 | 내용 | 용도 |
|---|---|---|
| `text` | 순수 본문 텍스트 | LLM 컨텍스트 전달용 |
| `text_with_meta` | `[발주기관: X \| 사업명: Y]\n본문` | 임베딩 벡터 생성용 |

임베딩 시 `text_with_meta`를 입력으로 사용함으로써, 벡터 공간 내에서 동일 기관의 청크들이 자연스럽게 클러스터를 형성하게 된다. 이 접두사는 전체 텍스트의 약 5%를 차지하며, 임베딩 모델이 기관/사업 정보를 의미적으로 인코딩하는 데 충분한 신호를 제공한다.

## 5.2 의도 추론 기반 적응형 하이브리드 검색 오케스트레이션 (Retriever)

### 왜 단순 벡터 검색으로는 부족한가

2장에서 확인한 Naïve RAG의 한계 중 가장 치명적인 것은 **기관 혼동**과 **고유명사 검색 실패**였다. Dense 검색만으로는 "국민연금공단의 2024년 이러닝 사업 예산"이라는 질문에서 '국민연금공단'과 '2024년'이라는 정확한 식별자를 구분하지 못하고, 유사한 맥락의 타 기관 데이터를 가져올 수 있다.

이 문제를 체계적으로 해결하기 위해, 본 시스템의 Retriever는 단순한 검색 모듈이 아니라 **7단계의 의사결정 파이프라인을 오케스트레이션하는 지휘자**로 설계되었다.

### 검색 오케스트레이션 7단계 상세 분석

#### 제1단: 내재 의도 스캐닝 및 메타데이터 필터 자동 추출

사용자 질문이 입력되면, `src/bidmate_rag/retrieval/filters.py`의 `extract_metadata_filters` 함수가 즉시 작동하여 질문 내부의 의도를 해부한다. 이 함수는 다음과 같은 계층적 필터 추출 로직을 실행한다.

**기관명 추출 우선:** 먼저 전체 발주기관 목록(`agency_list`)과 질문을 교차 대조하여 기관명을 추출한다. 기관이 1개 감지되면 즉시 `{"발주 기관": "감지된기관"}` 필터를 반환한다. 기관이 2개 이상 감지되면 비교형 질문으로 판단하여 단일 필터를 적용하지 않고 팬아웃(Fan-out) 검색으로 분기시킨다.

**도메인 키워드 추출:** 기관명이 감지되지 않으면, 질문에서 도메인 키워드를 스캔한다. 12개 도메인 카테고리(교육/학습, 안전/재난, 웹/포털, AI/데이터 등)에 대한 판별 키워드 사전이 정의되어 있으며, 매칭되는 도메인이 있으면 `{"사업도메인": "매칭도메인"}`으로 필터를 생성한다.

**기관유형 추출:** 도메인도 매칭되지 않으면, "대학교", "공기업", "지자체" 등의 기관유형 키워드를 탐색한다.

**범위 필터 병합:** 별도로 `extract_range_filters` 함수가 "10억 이상", "2024년" 등의 금액 범위와 연도 조건을 정규표현식으로 추출하여 Where 절에 추가한다.

**회피 키워드 감지:** 질문에 "다른 기관", "외에", "말고" 등의 키워드가 포함되면, 현재 문맥의 기관을 의도적으로 배제하기 위해 필터를 적용하지 않는다. 이 섬세한 예외 처리가 없으면, "국민연금공단 말고 다른 기관의 사례는?"이라는 질문에 오히려 국민연금공단만 필터링하는 역설이 발생한다.

#### 제2단: 멀티턴 대화 기반 맥락 상속 (History-aware Query Rewrite)

후속 질문에서 "그 사업의 예산은?"과 같이 대명사가 사용되면, 시스템은 이전 대화 맥락에서 지시 대상을 복원해야 한다. `src/bidmate_rag/retrieval/multiturn.py`에 구현된 쿼리 재작성 엔진이 이 역할을 수행한다.

재작성 엔진은 3가지 모드를 지원한다.

| 모드 | 작동 방식 | 용도 |
|---|---|---|
| `llm_with_rule_fallback` | LLM으로 재작성 시도 → 실패 시 규칙 기반 폴백 | 기본 설정 |
| `rule_only` | LLM 없이 규칙 기반만 사용 | 비용 절감, 로컬 환경 |
| `llm_only` | LLM 재작성만 사용 | 최고 품질 추구 |

**LLM 기반 재작성의 핵심 설계:**

LLM 재작성 시, 시스템은 단순히 "이전 대화를 참고해 질문을 다시 써줘"라고 요청하지 않는다. 정교하게 설계된 프롬프트가 다음과 같은 제약 조건을 LLM에 부과한다.

- 재작성된 질문 **한 줄만** 출력할 것
- 이전 assistant 답변에만 있던 **사실을 새로 가져오지 말 것**
- 현재 질문이 이미 충분히 분명하면 **원문 그대로 반환**할 것
- `section_hint`로 RFP 문서 내 해당 질문이 묻는 **섹션명을 추출**할 것

이 `section_hint`는 이후 규칙 기반 Soft Boost의 섹션 부스팅에 활용되는 핵심 신호다. 예를 들어 "보안 요구사항이 뭐야?"라는 질문에서 `section_hint: "보안 요구사항"`이 추출되면, Soft Boost 엔진이 "보안" 섹션 헤더를 가진 청크에 가산점을 부여한다.

**규칙 기반 재작성의 폴백 로직:**

LLM 호출이 실패하거나 비용을 절감해야 하는 경우, 규칙 기반 재작성이 동작한다. 이 로직은 "그 사업", "이 문서", "해당 기관" 등 12개의 후속 질문 키워드 패턴을 탐지하여, 이전 대화에서 추출한 주제어/기관명으로 치환한다.

| 원본 질문 | 재작성 결과 |
|---|---|
| "그 사업의 예산은?" | "국민연금공단 이러닝 사업의 예산은?" |
| "이 기관의 다른 요구사항은?" | "한국수출입은행의 다른 요구사항은?" |

**대화 메모리 구조 (Slot Memory):**

단순 대화 이력 참조를 넘어, 시스템은 `ConversationMemory` 모듈을 통해 구조화된 **슬롯 메모리(Slot Memory)**를 관리한다.

| 메모리 구성 요소 | 역할 | 설정값 |
|---|---|---|
| `summary_buffer` | 오래된 대화 턴을 자연어로 요약 저장 | 최대 400자, 최근 4턴은 원문 유지 |
| `slot_memory` | 기관명, 사업명 등 핵심 엔티티를 키-값 쌍으로 유지 | 활성화 |

슬롯 메모리에 `{"기관": "국민연금공단", "사업": "이러닝 시스템 구축"}`이 기록되어 있으면, 후속 질문의 재작성 시 이 정보가 프롬프트에 주입되어 맥락 복원 정확도가 향상된다.

**Timeout과 비용 제어:**

멀티턴 재작성의 LLM 호출에는 `timeout_seconds: 30`과 `max_completion_tokens: 16000` 제한이 설정되어 있다. 30초 내에 응답이 오지 않으면 즉시 규칙 기반 폴백으로 전환하여, 사용자가 무한 대기에 빠지는 것을 방지한다. 모든 재작성 호출의 토큰 사용량과 비용은 자동으로 추적되어 실험 로그에 기록된다.

#### 제3단: 팬아웃(Fan-out) 검색 — 비교형 질문의 균형 잡힌 증거 확보

"국민연금공단과 한국수출입은행의 보안 요구사항을 비교해줘"와 같은 비교형 질문을 일반적인 단일 검색으로 처리하면, 한쪽 기관의 관련 문서가 상위권을 독점하여 비교 분석이 불가능해진다.

본 시스템은 `should_fan_out_multi_source_query` 함수를 통해 비교형 질문을 감지한다. 감지 로직은 다음과 같다.

| 감지 조건 | 키워드 예시 |
|---|---|
| 비교/대조 키워드 존재 | "비교", "차이", "차액", "합산", "공통", "대비" |
| 열거 의도 키워드 존재 | "각각", "순서대로" |
| 다기관 힌트 키워드 존재 | "각 기관", "기관별", "두 기관", "사업별" |

비교형 질문으로 판정되면, Retriever는 `_build_scoped_filters` 메서드를 통해 Where 필터를 기관별로 분해하고, **각 기관에 대해 독립된 검색 파이프라인을 병렬 실행**한다.

| 단계 | 처리 |
|---|---|
| ① 필터 분해 | `{"발주 기관": {"$in": ["A", "B"]}}` → `[{"발주 기관": "A"}, {"발주 기관": "B"}]` |
| ② 독립 검색 | A기관 검색 결과 리스트, B기관 검색 결과 리스트 생성 |
| ③ 라운드 로빈 병합 | A 결과 1개 → B 결과 1개 → A 결과 1개 → ... 교대 배치 |
| ④ 다양성 보장 | `seen_doc_ids`로 동일 문서 독점 방지 |

라운드 로빈 병합의 핵심은 **새로운 문서를 우선 수집(prefer_new_docs)**하는 2패스(Two-pass) 알고리즘이다. 첫 번째 패스에서는 아직 등장하지 않은 문서의 청크만 수집하고, 부족하면 두 번째 패스에서 이미 등장한 문서의 청크도 허용한다. 이를 통해 A기관 100개 청크와 B기관 50개 청크가 있을 때, A기관이 상위권을 독점하는 현상을 구조적으로 방지한다.

#### 제4단: 하이브리드 교차 검색 풀 구축 (Dense + Sparse + RRF)

필터가 적용된 검색 공간 내에서, **Dense(임베딩 벡터) 검색**과 **Sparse(BM25 키워드) 검색**이 동시에 작동한다. `src/bidmate_rag/retrieval/hybrid.py`에 구현된 하이브리드 검색 로직이 이 과정을 관장한다.

**Dense 검색의 강점과 약점:**

| 강점 | 약점 |
|---|---|
| 의미적 유사성 포착 ("예산" ↔ "사업비") | 고유명사 정확 매칭 실패 ("Qwen2.5" 검색 불가) |
| 동의어·유의어 대응 | 숫자·코드 구분 불가 (모든 숫자가 비슷하게 인코딩) |
| 문맥 기반 추론 | 드물게 등장하는 전문 용어 약세 |

**Sparse (BM25) 검색의 강점과 약점:**

| 강점 | 약점 |
|---|---|
| 정확한 키워드 매칭 ("지체상금률" 정확 검색) | 동의어 대응 불가 ("예산" ≠ "사업비") |
| 숫자·고유명사 완벽 검색 | 문맥 이해 없음 |
| 빈도 기반 중요도 판별 | 일반적인 단어에 과도한 가중치 부여 가능 |

두 검색 엔진의 사각지대가 정확히 상호 보완적이다. 이것이 하이브리드 검색의 존재 이유다.

**후보 풀(Pool) 크기 결정 로직:**

`resolve_hybrid_pool_sizes` 함수는 최종 Top-K에서 역산하여, Dense와 Sparse 각각이 몇 개의 후보를 가져올지 결정한다.

| 조건 | Dense Pool 크기 | Sparse Pool 크기 |
|---|---|---|
| 기본 설정 (현재 채택) | `Top-K × dense_pool_multiplier` | `Top-K × sparse_pool_multiplier` |
| CE 활성화 시 (미채택) | `Top-K × 4` | `Top-K × dense_pool_multiplier` |
| Sparse 비활성 시 | `Top-K × 4` | 0 |

현재 시스템의 기본 설정(`configs/retrieval.yaml`)에서 `dense_pool_multiplier: 3`, `sparse_pool_multiplier: 3`으로 설정되어 있으므로, Top-K가 5일 때 Dense는 최소 15개, Sparse도 15개의 후보를 가져온다. 코드상으로는 Cross-Encoder(CE) 활성화 시 Dense 풀을 `Top-K × 4`(20개)로 확대하는 로직이 구현되어 있으나, CE가 전략적으로 제외되어(5.4절 참조) 현재는 3배수 설정이 적용된다.

**Reciprocal Rank Fusion (RRF) — 두 세계의 융합:**

Dense와 Sparse 각각의 결과 리스트가 생성되면, `reciprocal_rank_fusion` 함수가 이들을 하나의 통합 랭킹으로 융합한다. RRF 알고리즘의 수학적 원리는 다음과 같다.

각 청크의 RRF 점수는 해당 청크가 각 검색 결과에서 차지하는 순위(rank)를 기반으로 계산된다.

> **RRF_score(chunk) = Σ 1/(k + rank_i)**

여기서 `k`는 RRF 상수(기본값 60)이며, 이 값이 클수록 순위 차이가 점수에 미치는 영향이 완만해진다. 즉, 1등과 2등의 점수 차이가 줄어든다.

현재 시스템의 설정: `rrf_k: 60`

RRF 점수가 계산된 후, 상위 N개를 선택하고 Min-Max 정규화를 적용하여 0~1 범위로 변환한다. 이 정규화된 점수가 각 청크의 `score` 필드에 기록된다.

또한, 디버깅 및 분석을 위해 각 청크의 메타데이터에 `hybrid_dense_score`, `hybrid_sparse_score`, `hybrid_rrf_score` 값이 자동으로 기록된다. 이를 통해 특정 청크가 Dense에서 높았는지 Sparse에서 높았는지, 아니면 양쪽 모두에서 높았는지를 사후 분석할 수 있다.

#### 제5단: 재작성 쿼리와 원본 쿼리의 결과 병합

질의 재작성이 적용된 경우, 시스템은 재작성된 쿼리와 원본 쿼리 모두로 검색을 수행하고 결과를 병합한다. 이는 재작성이 의도를 왜곡할 위험에 대한 방어 장치다.

`_merge_query_variant_results` 메서드는 두 결과 리스트를 `chunk_id` 기준으로 합치되, 동일 청크가 양쪽에 모두 존재하면 더 높은 점수를 채택한다. 이 "최대 점수 채택(Max-Score Selection)" 전략은 재작성의 이점을 살리면서도 원본 쿼리의 보수적 검색 결과를 버리지 않는 균형을 달성한다.

#### 제6단: Fallback 체인 — 검색 실패 시 자동 완화

검색 결과가 0건인 경우, 시스템은 다단계 Fallback 로직을 실행한다.

| Fallback 단계 | 조건 | 동작 |
|---|---|---|
| ① `where_document` 제거 | 본문 필터 적용 후 0건 | 본문 필터만 해제하고 재검색 |
| ② 확장 필터로 완화 | `base_where ≠ where` 이고 0건 | 자동 추가된 문서 필터를 제거하고 기본 필터로 재검색 |

이 Fallback 체인은 "필터가 너무 엄격해서 정답이 잘려나가는" 상황을 자동으로 복구한다.

#### 제7단: 글로벌 1차 랭킹 출력

위 모든 단계를 거친 후보 청크 리스트가 규칙 기반 Soft Boost 재정렬 모듈(5.4절)로 전달된다.

## 5.3 5-Tier 구조 기반 초정밀 프롬프트 엔지니어링 및 페르소나 주입

### 프롬프트가 RAG 시스템의 성패를 가르는 이유

검색된 컨텍스트가 아무리 정확해도, LLM에 전달하는 프롬프트(System Prompt)의 설계가 부실하면 다음과 같은 실패가 발생한다.

| 프롬프트 결함 | 결과적 실패 |
|---|---|
| "모르면 모른다고 해"라는 지시 부재 | 컨텍스트에 없는 내용을 환각으로 생성 |
| 출력 언어 미지정 | 한국어 질문에 영어로 응답 |
| 출력 포맷 미지정 | 구조화되지 않은 장문의 산문 출력 |
| 근거 명시 미요구 | 출처 불명의 답변 생성 |

### 5-Tier 프롬프트 아키텍처

본 시스템은 프롬프트를 5개 계층(Tier)으로 구조화하여, 각 계층을 독립적으로 실험하고 최적화할 수 있도록 설계하였다.

| Tier | 명칭 | 역할 | 예시 |
|---|---|---|---|
| **Tier 1** | Persona (P) | LLM의 역할 정체성 부여 | "당신은 공공입찰 전문 컨설턴트입니다" |
| **Tier 2** | Constraints (C) | 행동 제약 조건 | "컨텍스트에 없는 내용은 '확인되지 않습니다'로 답하세요" |
| **Tier 3** | Format (F) | 출력 형식 규칙 | "마크다운 형식으로, 출처를 반드시 명시하세요" |
| **Tier 4** | Few-shot (S) | 모범 답변 예시 | 질문-답변 쌍 1~3개 제시 |
| **Tier 5** | Chain-of-Thought (T) | 추론 과정 유도 | "먼저 관련 컨텍스트를 식별하고, 그 후 답변을 구성하세요" |

각 Tier는 YAML 설정 파일(`configs/prompts/prompt_P{n}_C{n}_F{n}_S{n}_T{n}.yaml`)로 버전 관리되며, **프롬프트의 어떤 계층을 변경했을 때 성능이 어떻게 변하는지를 통제 변인 실험으로 분리 측정**할 수 있다.

### 프롬프트 ID 체계와 실험 관리

프롬프트는 `P{n}_C{n}_F{n}_S{n}_T{n}` 형식의 고유 ID로 버전 관리된다. 예를 들어:

| 프롬프트 ID | 의미 |
|---|---|
| P1_C1_F1_S1_T1 | Persona v1 + Constraints v1 + Format v1 + Shot v1 + CoT v1 |
| P2_C1_F1_S1_T1 | Persona만 v2로 변경, 나머지 동일 |
| P1_C2_F1_S1_T1 | Constraints만 v2로 변경, 나머지 동일 |

이 체계를 통해 **정확히 어떤 프롬프트 요소가 Hallucination 발생률의 급격한 변동을 야기하는지** 인과 관계를 추적할 수 있다.

### 핵심 Constraints 설계: 환각 방어의 최전선

Tier 2 (Constraints)는 RAG 시스템에서 가장 중요한 프롬프트 계층이다. 이 계층의 핵심 지시 사항들은 다음과 같다.

| 제약 조건 | 목적 | 효과 |
|---|---|---|
| "컨텍스트에 없으면 '확인되지 않습니다'로 답하라" | 환각 방지 | LLM이 모르는 것을 인정하도록 강제 |
| "반드시 한국어로 답변하라" | 언어 일관성 | 영문 혼용 방지 |
| "출처 문서명과 섹션을 명시하라" | 근거 투명성 | 사용자가 답변의 근거를 확인 가능 |
| "추측이나 외부 지식을 사용하지 마라" | 근거주의 강화 | 컨텍스트 밖의 정보 유입 차단 |

## 5.4 규칙 기반 Soft Boost 재정렬 시스템 및 Cross-Encoder 전략적 제외 분석

### 최종 채택 아키텍처: 규칙 기반 Soft Boost 재정렬

5.2절의 하이브리드 검색 오케스트레이션이 출력한 후보 청크 리스트는, 최종 LLM 전달 전에 **규칙 기반 Soft Boost 재정렬** 단계를 거친다. 이 단계는 RRF 융합 점수에 도메인 특화 휴리스틱 가중치를 가산하여, RFP 문서 구조에 최적화된 최종 랭킹을 산출한다.

본 시스템의 코드베이스에는 Cross-Encoder(CE) 리랭킹 파이프라인이 완전히 구현되어 있으나(commit `50f0028`, 2026-04-13), **사전 설계 분석을 통해 전략적으로 제외**하였다. 최종 배포 설정에서 `reranker_model: null`로 의도적으로 비활성화되어 있으며, 31개 전체 실험 Run에서 CE를 활성화한 사례는 0건이다. CE 제외의 상세 엔지니어링 근거는 본 절 후반부에서 서술한다.

| 비교 항목 | Bi-Encoder (1차 검색) | Soft Boost (최종 채택) | Cross-Encoder (구현됨, 미채택) |
|---|---|---|---|
| 재정렬 방식 | 벡터 유사도 | 규칙 기반 가중치 가산 | 양방향 어텐션 스코어링 |
| 추가 모델 필요 | ✅ 임베딩 모델 | ❌ 모델 불필요 | ✅ CE 전용 모델 (~1.2GB VRAM) |
| 추가 Latency | — | **< 1ms** | **200~500ms** (GPU) / **2~5초** (CPU) |
| VRAM 부담 | 기존 임베딩 공유 | **0GB 추가** | **~1.2~3.2GB 추가** |
| RFP 도메인 특화 | 범용 | ✅ 섹션/테이블/기관 규칙 내장 | 범용 (도메인 파인튜닝 필요) |
| 시나리오 A/B 동시 호환 | ✅ | ✅ | ❌ (시나리오 A에서 VRAM 부족 리스크) |

### Soft Boost 처리 파이프라인

`src/bidmate_rag/retrieval/reranker.py`에 구현된 최종 재정렬 파이프라인은 다음과 같은 단계를 수행한다. Cross-Encoder 스코어링 단계는 `reranker_model: null` 설정에 의해 자동으로 건너뛰어지며(`cross_encoder_rerank` 함수의 `if not reranker: return results` 가드), 규칙 기반 Soft Boost만이 실제 재정렬을 수행한다.

#### 제1단: 텍스트 정규화 기반 교차 검증

`_normalize_text` 함수는 메타데이터 매칭 시 특수문자, 대소문자, 파일 확장자 차이로 인한 오탐을 방지한다. 이 정규화를 통해 "국민연금공단.hwp"와 "국민연금공단"이 동일한 엔티티로 매칭된다. `build_reranker_text` 함수 역시 발주기관과 사업명을 접두사로 결합하여 메타데이터 매칭의 정확도를 높인다.

입력 형식:
```
[발주기관: 국민연금공단 | 사업명: 이러닝 시스템 구축]
제안업체는 1,000TPS 이상의 처리 성능을 보장해야 하며...
```

이 접두사를 통해 Cross-Encoder가 "이 청크가 어떤 기관의 어떤 사업에 대한 것인지"를 인지한 상태에서 관련성을 판단하게 된다. 질문이 "국민연금공단의 성능 요구사항"이라면, 접두사의 기관명이 일치하는 청크에 자연스럽게 높은 점수가 부여된다.

#### [전략적 제외] Cross-Encoder 스코어링 단계 — 비활성화됨

`cross_encoder_rerank` 함수는 코드에 완전히 구현되어 있으나, `reranker_model: null` 설정에 의해 **현재 비활성화 상태**다. 함수 내부의 `if not reranker or not results: return results` 가드가 즉시 원본 리스트를 반환하므로, CE 스코어링은 실행되지 않는다.

코드에 등록된 CE 모델 후보 (향후 활성화 가능):

| 모델 | 특성 | VRAM 요구량 |
|---|---|---|
| `dragonkue/bge-reranker-v2-m3-ko` | 한국어 특화 Cross-Encoder | ~1.2GB |
| `BAAI/bge-reranker-v2-m3` | 다국어 Cross-Encoder | ~1.2GB |
| `Qwen3-Reranker-0.6B` | 경량 Reranker | ~0.4GB |

CE 활성화는 `configs/retrieval.yaml`에서 `reranker_model` 값을 변경하는 것만으로 가능하며, 코드 수정이 불필요하다. 그러나 아래 3가지 설계 분석에 의해 최종 배포에서 전략적으로 제외하였다.

#### 제2단: 구조적 파생 변수 인센티브 가중치 부여 (Multi-factor Soft Boost)

`_normalize_text` 함수는 메타데이터 매칭 시 특수문자, 대소문자, 파일 확장자 차이로 인한 오탐을 방지한다. 정규화 과정은 다음과 같다.

| 정규화 단계 | 처리 |
|---|---|
| ① | 소문자 변환 |
| ② | `.pdf`, `.hwp`, `.hwpx`, `.docx`, `.doc` 확장자 제거 |
| ③ | 특수문자·언더스코어 제거, 공백 제거 |

이 정규화를 통해 "국민연금공단.hwp"와 "국민연금공단"이 동일한 엔티티로 매칭된다.

이것이 본 시스템의 **최종 재정렬 메커니즘**이다.

`rerank_with_boost` 함수는 RRF 융합 점수에 추가적인 **규칙 기반 휴리스틱 가중치**를 산술적으로 합산하여 최종 순위를 결정한다. Cross-Encoder가 비활성화된 현재 설정에서, 이 Soft Boost가 유일한 재정렬 단계로서 기능한다.

| 부스팅 유형 | 가산 점수 | 발동 조건 |
|---|---|---|
| **섹션 부스팅** | +0.12 | 질의의 `section_hint`가 청크의 섹션 헤더 또는 본문에 포함 |
| **테이블 부스팅** | +0.08 | 질문에 "요구사항", "배점", "예산" 등 테이블 키워드가 있고 청크가 `content_type: table` |
| **메타데이터 부스팅** | +0.12 | 질문에 포함된 기관명/사업명이 청크의 메타데이터와 정확 매칭 |
| **합산 상한선** | 0.15 | 모든 부스팅의 합계가 이 값을 초과하지 못함 (`max_total`) |

**MAX_TOTAL_BOOST 상한선 설계 근거:**

부스팅 가중치에 상한선이 없으면, 메타데이터가 완벽히 일치하는(기관명+사업명+섹션 헤더) 청크가 **본문 내용과 무관하게** 최상위에 올라올 수 있다. 예를 들어 국민연금공단의 "목차" 페이지가 기관명·사업명 매칭으로 높은 부스팅을 받아 실제 "보안 요구사항" 청크를 밀어내는 역설이 발생한다.

현재 시스템의 `configs/retrieval.yaml` 설정:

| 파라미터 | 설정값 |
|---|---|
| `boost.section` | 0.12 |
| `boost.table` | 0.08 |
| `boost.metadata` | 0.12 |
| `boost.max_total` | 0.15 |

**비교형 질문에서의 메타데이터 부스팅 비활성화:**

`is_comparison_query` 함수로 비교형 질문이 감지되면, 메타데이터 부스팅을 의도적으로 비활성화한다. "A기관과 B기관을 비교해줘"라는 질문에서 A기관 청크에만 메타데이터 부스팅이 적용되면, B기관 청크가 밀려나 공정한 비교가 불가능해지기 때문이다.

#### 제3단: 최종 Shortlist 확정 및 LLM 전달

모든 스코어링이 완료되면, 최종 상위 **Top-K개(기본 5개)**의 청크를 확정하여 LLM의 컨텍스트 프롬프트로 전달한다.

**Top-K = 5를 채택한 이유: "Lost in the Middle" 현상 방어**

| Top-K 설정 | 장점 | 단점 |
|---|---|---|
| 3 | 노이즈 최소화 | 정보 부족 위험 |
| **5 (채택)** | 정보량과 집중도의 균형 | — |
| 10 | 정보 다양성 확보 | 중간 위치 청크 무시 현상 발생 |
| 20+ | — | 레이턴시 4배 증가, MRR 이점 미미 |

"Lost in the Middle" 현상이란, LLM이 컨텍스트의 처음과 마지막 부분에 위치한 정보는 잘 활용하지만, **중간에 위치한 정보는 무시하는 경향**을 가리킨다. 10개 이상의 청크를 제공하면 6~8번째 청크의 핵심 정보가 답변에 반영되지 않는 현상이 관찰되었으며, 이를 방지하기 위해 5개로 엄격히 제한하였다.

### 전체 검색-재정렬 파이프라인 데이터 흐름 요약

| 단계 | 입력 규모 | 출력 규모 | 핵심 메커니즘 |
|---|---|---|---|
| ① 메타데이터 필터링 | 13,951개 전체 | 수백~수천 개 | Where Clause 하드 필터 |
| ② Dense 검색 | 수백~수천 개 | 15~20개 | 코사인 유사도 |
| ③ Sparse 검색 | 수백~수천 개 | 15개 | BM25 키워드 매칭 |
| ④ RRF 융합 | 30~35개 | 15~20개 | Reciprocal Rank Fusion |
| ⑤ Cross-Encoder | 15~20개 | 5개 | 양방향 어텐션 스코어링 |
| ⑥ 부스팅 | 5개 | 5개 | 섹션/테이블/메타데이터 가산 |
| ⑦ LLM 전달 | 5개 | 답변 1건 | 시스템 프롬프트 + 컨텍스트 |

13,951개의 청크에서 출발하여, 7단계의 필터링·스코어링·재정렬을 거쳐 **최종 5개의 정밀 타격 증거만을 LLM에 공급**한다. 이 엄격한 깔때기(Funnel) 구조가 BidMate RAG 시스템의 높은 Faithfulness(충실도)와 Context Precision(문맥 정밀도)을 달성하는 핵심 아키텍처다.
