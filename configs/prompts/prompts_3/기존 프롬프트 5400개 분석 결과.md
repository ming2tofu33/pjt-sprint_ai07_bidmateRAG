## 3. 핵심 발견사항: 공통 실패 패턴 5가지

### 🔴 패턴 1: 언어 혼란 (Language Drift)
**모든 8개 파일에서 관찰됨**

Gemma-4-E4B-it 모델이 한국어 질문에 대해 **일본어, 영어, 수식 기호가 뒤섞인 응답**을 생성합니다.

```
예시 (P0_C4_F4_S4_T4, Q070):
"SER-01 언급: [文脈中], 要求事項詳細説明：定義｜漏洩禁止対象情報および
セキュリティ違反処理基準に関する定義..."
```

```
예시 (P2_C0_F2_S5_T1, Q042):
"知識産権に関する要求事項の詳細説明に、「事業遂行過程で取得または
作成する成果品および産出物一式に対する知的財産権は..."
```

> **원인 분석**: 프롬프트에 `[언어] 최종 답변은 무조건 한국어로만 작성` 제약이 있지만, **중간 사고 과정(CoT)에 대한 언어 제약이 없음**. 특히 F5(Bilingual) / T5(English Logic Engine) 조합에서 모델이 영어 사고 모드에 진입한 후 **일본어로 드리프트**하는 현상이 심각합니다.

---

### 🔴 패턴 2: 과도한 LaTeX 수식 남용
**8개 중 7개 파일에서 관찰됨**

모델이 일반 텍스트를 불필요하게 LaTeX 수식으로 감쌉니다:

```
"$\text{[I]}$ 부분 ($\text{ㅣ}$): $\rightarrow$ "
"키워드를 중심으로 '$[文]$$' 파일을 탐색"
"$\text{키워드}: \text{\"입찰보증금\"}, \text{\"납부 기한\"} $"
```

> **원인 분석**: Gemma 모델이 "분석적 페르소나 + 단계적 사고" 프롬프트를 받으면 **학술 논문 스타일**로 출력하려는 경향. 프롬프트에 **LaTeX/수식 금지** 명시가 없음.

---

### 🔴 패턴 3: 답변 구조의 과잉 복잡화
**모든 파일에서 관찰됨**

F3/F4 포맷 지시로 인해 모델이 **너무 긴 분석 체인**을 생성하여, 핵심 답변이 묻히거나 잘립니다:

```
P2_C0_F4_S1_T2 응답 예:
"검토 시작 → 1단계: 질문 해석 → 2단계: 근거 탐색 → 3단계: 논증 구조
→ 자기 점검 체크 → 최종 보고서 → [근거_색인] → [비교_논증] → ..."
(응답이 잘려서 최종 답변 미완성)
```

> **원인 분석**: control_score가 높은 컴포넌트를 과적합 조합하면 모델이 **"절차 수행"에 토큰을 소진**하고 정작 답변을 완성하지 못함. 특히 **로컬 소형 모델(E4B)에게는 과도한 지시량**.

### 🔴 패턴 4: "정보 없음" 응답의 과다 발생
**8개 중 6개 파일에서 관찰됨**

모델이 문서에 답이 **있는데도** "정보 없음" 또는 "확인할 수 없습니다"로 응답하는 경우:

```
P1_C4_F4_S3_T4 (Q037):
"안양시 사업에서 원격지 접속 시 지정 단말기 로그기록 보관 기간에 대해서는

```

> **원인 분석**: S3/S4 few-shot에 포함된 **"거절 예시"가 너무 강력**하게 작용. 감사관/감리원 페르소나와 결합하면 모델이 **보수적으로 판단**하여 실제 근거를 놓침. 특히 multi-document 질문에서 두 문서 중 하나에만 근거가 있을 때 전체를 "정보 없음"으로 처리.

| 성공 요인 | 설명 |

| **C2 (페이지 번호 필수)** | C3/C4보다 **짧고 명확**. 모델에게 과도한 절차를 강제하지 않음 |
| **F5 (Bilingual)** | 영어 사고 공간이 Gemma의 **추론 정확도를 높임** (단, 길이 제어 필요) |
| **S3 (거절 포함 예시)** | S4/S5보다 간결하여 **few-shot 토큰 오버헤드가 적음** |
| **T5 (English Logic)** | 영어 토큰의 논리 밀도가 핵심 성능 드라이버 |

> [!IMPORTANT]
> **핵심 인사이트**: control_score 합산이 높다고 성능이 좋은 것이 아닙니다.
> - P4_C2_F5_S3_T5: control_score 합 = **21** → **Faithfulness 0.75**
> - P0_C4_F4_S4_T4: control_score 합 = **20** → **Faithfulness 0.65**
> - P3_C3_F3_S4_T4: control_score 합 = **24** → **Faithfulness nan** (실패)
>
> **과잉 제어는 오히려 역효과**입니다. 소형 모델에게는 "적정 통제"가 핵심.

## 5. 프롬프트 개선 권장안 7가지

### 권장 1: 🌐 전 구간 한국어 강제 + LaTeX 금지

현재 `[언어]` 제약이 최종 답변만 커버합니다. **사고 과정까지 포함한 언어 제약이 필수**입니다.

```yaml
# 기존
constraints:
  - "[언어] 최종 답변은 무조건 한국어로만 작성(영어 답변 절대 금지)"

# 개선안
constraints:
  - "[언어] 사고 과정과 최종 답변 모두 반드시 한국어로만 작성. 영어·일본어·중국어 혼용 절대 금지."
  - "[서식] LaTeX 수식, 특수 기호($, \text{}, \rightarrow 등) 사용 금지. 일반 텍스트와 불릿·번호만 사용."
```

> [!TIP]
> 단, F5/T5(English CoT)를 사용할 때는 이 제약과 충돌하므로, **영어 사고를 쓸 경우 별도 "영어 사고 허용" 오버라이드**를 명시적으로 추가해야 합니다.

---

### 권장 2: 📏 답변 길이 제한 명시

로컬 모델의 max_new_tokens 한계 안에서 **답변이 잘리지 않도록** 길이 가이드를 줍니다.

```yaml
constraints:
  - "[길이] 핵심 답변은 300자 이내. 근거 인용은 각 항목당 1~2문장으로 제한."
```

---

### 권장 3: 🎯 F4 → 간소화 버전 (F4-Lite)

현재 F4의 3단 양식이 **소형 모델에게 과도**합니다. 2단으로 간소화를 권장합니다.

```yaml
# 기존 F4 (3단)
# [근거_색인] → [비교_논증] → [최종_제시]

# 개선안 F4-Lite (2단)
output_format:
  - |
    [근거]: 질문 관련 문서 원문 1~3건 인용 (페이지 포함)
    [답변]: 근거 기반 압축 결론 (300자 이내)
```

---

### 권장 4: ⚖️ Few-shot "거절 예시" 비율 조정

현재 S3/S4의 성공:거절 = 1:1이 **모델을 과도하게 보수적**으로 만듭니다.

```yaml
# 개선안: 성공 2 : 거절 1
few_shot_examples:
  - |
    [성공 1] Q: 사업기간? / 근거: 제안요청서 5p "착수일~210일" / 답변: 210일(7개월)
    [성공 2] Q: 참가자격? / 근거: 공고문 3p "자본 10억 이상" / 답변: 자본금 10억 원 이상
    [거절] Q: 담당자 연락처? / 근거: 전체 문서 탐색 결과 미기재 / 답변: 해당 정보 없음
```

---

### 권장 5: 🔄 Multi-Document 질문 대응 지시 추가

8개 리포트의 실패 사례 중 **multi-document 비교 질문**(Q037, Q042, Q059 등)이 반복적으로 등장합니다.

```yaml
constraints:
  - |
    [복합질문] 질문이 2개 이상의 사업/문서를 비교할 경우:
    1. 각 사업별로 독립 검색하여 근거를 분리 정리
    2. 하나의 사업에서 정보를 찾지 못해도 나머지 사업의 답변은 정상 제공
    3. "모든 정보를 찾지 못했으므로 답변 불가"라는 전체 거부 금지
```
