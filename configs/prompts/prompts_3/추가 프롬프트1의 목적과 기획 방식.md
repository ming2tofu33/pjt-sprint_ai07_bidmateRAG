# 🛠️ 프롬프트 컴포넌트 재설계안 (Gemma-4-E4B 최적화)

> [!IMPORTANT]
> 이 문서는 Test A 실험 8건의 실패 패턴을 바탕으로, 각 프롬프트 컴포넌트(P/C/F/S/T)를 **소형 로컬 모델(Gemma-4-E4B)에 최적화**하여 재설계한 것입니다.
> 핵심 원칙: **"짧고 명확하게, 행동 지시 위주로, 금지 사항을 구체적으로"**

---

## 설계 철학: 기존 vs 개선

| 관점 | 기존 설계 | 개선 설계 |
|------|-----------|-----------|
| 지시량 | 많을수록 정확 | **소형 모델은 지시가 많으면 혼란** → 핵심만 |
| 언어 | "최종 답변만 한국어" | **전 과정 한국어 강제** (사고 포함) |
| 서식 | LaTeX/수식 허용 | **순수 텍스트만 허용** |
| 거절 | 성공:거절 = 1:1 | **성공:거절 = 2:1** (과보수 방지) |
| 출력 구조 | 3단 양식 | **2단 양식** (토큰 절약) |
| 길이 | 무제한 | **300자 핵심 답변 제한** |

## 01. 페르소나 (Persona) — 개선안

### ✅ 추천: P4-R (Revised) — "실용적 감리원"

```
기존 P4 문제점:
- "0.1%의 오차도 용납하지 않는" → 모델이 과도하게 보수적으로 변해 "정보 없음" 남발
- "질문에 바로 답하지 말고" → 불필요한 전수 조사에 토큰 소진
- 전체 길이가 너무 김 → 소형 모델의 지시 추종 능력 초과
```

**개선된 P4-R:**

```yaml
persona: |
  당신은 입찰 문서 전문 감리원입니다.
  - 반드시 제공된 문서 안의 사실만으로 답변합니다.
  - 문서에 근거가 있으면 핵심을 정확히 답변합니다.
  - 문서에 근거가 없으면 "해당 정보 없음"이라고만 답합니다.
  - 추측, 외부 지식, 개인 의견은 절대 사용하지 않습니다.
```

**변경 포인트:**
| 항목 | 기존 P4 | 개선 P4-R |
|------|---------|-----------|
| 캐릭터 강도 | "0.1% 오차 용납 불가" 극단적 | "전문 감리원" 균형적 |
| 행동 원칙 | 4줄 (신념/행동/검산 별도) | **4개 불릿**으로 압축 |
| 토큰 수 | ~120 토큰 | **~60 토큰** (50% 절감) |
| 핵심 차이 | "답하지 말고 목록화" | **"근거 있으면 답변, 없으면 없음"** |

> [!TIP]
> P4-R의 핵심은 "찾으면 답하라"는 **적극적 응답 유도**입니다. 기존 P4는 "답하기 전에 목록화하라"여서 모델이 목록 작성에만 집중하고 답변을 놓쳤습니다.

## 02. 제약 조건 (Constraints) — 개선안

### ✅ 추천: C2-R (Revised)

```
기존 문제점 정리:
- C0: 제약 없음 → 모델 자유도 과잉, 환각 발생
- C3: [철칙][필수][제약] 3단 → 지시가 길어서 소형 모델이 일부만 따름
- C4: 매뉴얼형 → 가장 긴 제약, Gemma가 소화 못함 (= nan 결과)
- C2: "외부지식 금지 + 페이지 번호" → 짧고 효과적 (최고 성능 조합의 C)
```

**개선된 C2-R:**

```yaml
constraints:
  - |
    [언어] 사고 과정·근거·최종 답변 모두 한국어로만 작성.
    영어, 일본어, 중국어 사용 금지.
  - |
    [서식] LaTeX 수식($, \text{} 등), 특수 기호(→, ∙ 등) 사용 금지.
    일반 텍스트, 불릿(-, ○), 번호(1. 2. 3.)만 허용.
  - |
    [근거] 답변의 모든 사실에는 문서 번호([1], [2] 등)를 표기.
    문서에 없는 정보는 절대 생성하지 않음.
  - |
    [길이] 핵심 답변은 300자 이내로 작성.
```

**변경 포인트:**
| 항목 | 기존 C4 | 개선 C2-R |
|------|---------|-----------|
| 길이 | ~180 토큰 | **~80 토큰** |
| 언어 제약 | 최종 답변만 | **전 과정** |
| LaTeX | 언급 없음 | **명시적 금지** |
| 길이 제한 | 없음 | **300자** |
| 절차 강제 | "사실 고립 → 논리 검증 → 최종 선별" | **없음** (F/T에게 위임) |

> [!WARNING]
> 기존 C3/C4는 C 안에서 "추출 → 대조 → 선별" 절차까지 지시했습니다. 이는 F(포맷)·T(추론)와 **역할이 중복**되어 모델에게 같은 말을 3번 하는 셈입니다. 개선안에서는 **C는 "하지 마라(금지)"에 집중**, **"이렇게 하라(절차)"는 F와 T에게만** 맡깁니다.

## 03. 출력 형식 (Format) — 개선안

### ✅ 추천: F4-Lite — "2단 간소 양식"

```
기존 문제점:
- F3: [섹션1_분석] → [섹션2_최종] → 이것만으로도 모델이 분석에 토큰 과소비
- F4: [근거_색인] → [비교_논증] → [최종_제시] → 3단이라 더 심각
- F5: <thought_en> → <answer_ko> → 영어 구간이 길어 한국어 답변 잘림
```

**개선된 F4-Lite:**

```yaml
output_format: |
  아래 양식을 반드시 지켜서 답변하세요:

  [근거]
  - 관련 문서 원문 인용 1~3건 (문서 번호 포함)

  [답변]
  - 근거 기반 핵심 결론 (300자 이내)
  - 문서에 정보가 없으면: "해당 정보 없음"
```

**변경 포인트:**
| 항목 | 기존 F4 (3단) | 개선 F4-Lite (2단) |
|------|---------------|-------------------|
| 단계 수 | 3단 (색인→논증→제시) | **2단 (근거→답변)** |
| "비교_논증" | 있음 (토큰 많이 소모) | **제거** (T에서 내부적으로 처리) |
| 누락 경고 | "*누락 시 오답 처리" | **삭제** (위협적 지시는 역효과) |
| 토큰 예산 | ~80 토큰 | **~40 토큰** |

---

### 대안 포맷 비교

```yaml
# F-Minimal: 최소 형식 (가장 가벼운 옵션)
output_format: |
  근거를 먼저 인용한 후, 핵심 답변을 한 문장으로 작성하세요.

# F4-Lite: 2단 간소 양식 (권장 — 위에 서술)

# F-Bilingual-Safe: 영어 CoT 안전 버전
output_format: |
  <analysis>
  List key facts from documents in 3 bullet points max. (English OK)
  </analysis>

  <answer>
  한국어 핵심 답변 (300자 이내, 근거 문서 번호 포함)
  </answer>
```

> [!TIP]
> F-Bilingual-Safe는 기존 F5의 문제를 해결합니다. `<thought_en>`이 무한정 길어지던 것을 **"3 bullet points max"로 영어 구간을 제한**합니다.

## 04. 예시 (Few-shot) — 개선안

### ✅ 추천: S3-R — "2성공 + 1거절"

```
기존 문제점:
- S1: 예시가 너무 짧아 패턴 학습 불충분
- S3/S4: 성공:거절 = 1:1 → 모델이 "거절하는 게 안전"이라고 학습
- S5: 영어 Logic 예시가 길어서 토큰 낭비, 실패시 언어 혼란 유발
```

**개선된 S3-R:**

```yaml
few_shot_examples:
  - |
    [예시 1 - 단답형]
    Q: 사업기간은 며칠입니까?
    [근거] [1] "사업기간: 착수일로부터 210일(7개월)" 확인
    [답변] 210일(7개월)입니다.

    [예시 2 - 비교형]
    Q: A사업과 B사업 모두 지식재산권을 발주처 단독 소유로 규정합니까?
    [근거]
    - [1] A사업: "지식재산권은 발주기관과 계약상대자가 공동 소유" 확인
    - [3] B사업: "지식재산권은 공동 소유를 원칙" 확인
    [답변] 아닙니다. 두 사업 모두 발주처 단독 소유가 아닌 공동 소유를 원칙으로 규정하고 있습니다.

    [예시 3 - 정보 없음]
    Q: 담당자 연락처는?
    [근거] 전체 문서([1]~[5]) 탐색 결과, 담당자 연락처 관련 기재 없음
    [답변] 해당 정보 없음.
```

**핵심 변경:**
| 항목 | 기존 S3/S4 | 개선 S3-R |
|------|-----------|-----------|
| 비율 | 성공 1 : 거절 1 | **성공 2 : 거절 1** |
| 예시 유형 | 단답형만 | **단답형 + 비교형 + 거절** |
| 길이 | 각 예시 2~3줄 | **각 예시 3~4줄** (포맷 시연) |
| 핵심 추가 | - | **비교형 예시** (multi-doc 질문 대응) |

> [!IMPORTANT]
> **[예시 2 - 비교형]이 가장 중요한 추가 사항**입니다. 실험 8건에서 반복 실패한 Q037, Q042 등은 모두 "A사업과 B사업을 비교하라"는 multi-document 질문이었습니다. 이런 패턴의 예시를 주면 모델이 **각 문서별로 분리 검색하는 법**을 학습합니다.

## 05. 추론 유도 (Thought Trigger / CoT) — 개선안

### ✅ 추천: T4-R — "3단 간결 추론"

```
기존 문제점:
- T1: "Think step by step" → 너무 모호, Gemma에서 효과 미미
- T3: "전수조사 → 검증 → 답변" → 방향은 맞지만 "전수조사"가 과도한 나열 유발
- T4: 리스팅/필터링/확정 3단 상세 매뉴얼 → 좋은 의도지만 너무 길음
- T5: English Logic Engine → 언어 드리프트 리스크
```

**개선된 T4-R:**

```yaml
thought_trigger: |
  다음 3단계를 내부적으로 수행한 후 답변하세요:
  1. 검색: 질문 핵심 키워드로 문서에서 관련 문장 1~3개 찾기
  2. 검증: 찾은 문장이 질문의 요구사항에 정확히 부합하는지 확인
  3. 답변: 검증된 정보만으로 [근거]+[답변] 양식에 맞게 작성
```

**변경 포인트:**
| 항목 | 기존 T4 | 개선 T4-R |
|------|---------|-----------|
| "전수조사/리스팅" | "누락 없이 나열" | **"1~3개 찾기"** (범위 제한) |
| "필터링" | "모순점 상세 대조" | **"부합하는지 확인"** (간결) |
| "확정" | "검증 통과 수치만" | **[근거]+[답변] 양식** (F와 연결) |
| 길이 | ~60 토큰 | **~40 토큰** |

> [!TIP]
> "1~3개 찾기"로 범위를 제한하는 것이 핵심입니다. 기존 "전수조사"는 모델이 문서의 모든 수치를 나열하느라 토큰을 소진했습니다. 소형 모델에게는 **"적게, 정확하게"**가 낫습니다.
