## 1. 파서: kordoc 선정

- **결정**: kordoc (Node.js CLI)을 메인 파서로 사용
- **배경**: RFP 100건 중 HWP 96건. CSV 텍스트 컬럼은 원본의 3~5% 수준 (21건이 1,000자 미만)
- **대안 검토**:

| 파서 | 언어 | 글자수 (국민연금) | 표 처리 | 구조 보존 |
|---|---|---|---|---|
| **kordoc** | Node.js | 81,459자 | 마크다운 표 | # 헤딩 + 표 |
| hwp-hwpx-parser | Python | 67,392자 | TableData 객체 | 목차 인코딩 깨짐 |
| CSV 텍스트 | - | 2,346자 | 없음 | 평문 |

- **선정 이유**: 마크다운 출력이 정제/청킹/LLM 입력 모든 단계에서 효율적. Node.js 의존성은 subprocess 3줄로 해결.
- **참조**: 02_preprocessing.ipynb, 섹션 12

## 2. 정제: 6종 노이즈 제거

- **결정**: `<br>` 태그, 중복 셀, 연속줄바꿈, 연속공백, PUA 문자, NBSP, kordoc Warning 제거
- **배경**: 9개 문서 사전 조사에서 `<br>` 17,502개, 중복 셀 2,098개 등 발견
- **핵심 판단**:
  - 표 행 내부 연속공백은 **보존** (마크다운 표 구조 유지)
  - 한국어 특수문자 (□○※①②③) **보존** (RFP 구조적 의미)
  - 중복 셀은 삭제가 아닌 **축소** (병합 셀 정보 유지)
- **결과**: 총 746,144자 제거 (평균 6.7%), 제거율은 문서 크기와 무관
- **참조**: 03_cleaning.ipynb, 섹션 14

## 3. 청킹: h1 분리 + 500자 병합

- **결정**: MarkdownHeaderTextSplitter로 h1만 분리 후, 500자 미만 섹션을 다음 섹션과 병합
- **배경**: 초기 구현에서 h1/h2/h3 모두 분리 → 34,983개 청크, 중앙값 81자, 56%가 100자 미만
- **대안**: h1/h2/h3 전부 분리, 고정 크기만 사용
- **수정 효과**:

| 지표 | 수정 전 | 수정 후 |
|---|---|---|
| 총 청크 | 34,983 | 14,035 (-60%) |
| 중앙값 | 81자 | 976자 (12배) |
| 100자 미만 | 56% | 1.0% |

- **참조**: 04_chunking.ipynb, 섹션 9, 13

## 4. 청크 크기: 1,000자 + 150자 오버랩

- **결정**: chunk_size=1000, chunk_overlap=150
- **배경**: 한국어는 영어 대비 토큰 2.3배 소모 → 1,000자 ≈ 500토큰
- **실험**: 500/800/1000/1500 × overlap 0/150 = 8가지 조합 비교
- **선정 이유**: RFP 분석형 질문에 적합한 밸런스. 표는 별도 보존 (max_table_size=1500)
- **참조**: 04_chunking.ipynb, 섹션 4

## 5. 메타데이터 보강

- **결정**: 기관유형, 사업도메인, 기술스택, 공개연도 4개 필드 추가
- **배경**: 14개 질문 시나리오 분석 결과, 사업도메인(21%), 기관유형(14%), 기술스택(14%)이 필터로 필요
- **분류 규칙**: v1 → v2 보강 ('기타' 21건→3건, 도메인 '기타' 56건→3건)
- **참조**: 04_chunking.ipynb, 섹션 1-1

## 6. 임베딩: text-embedding-3-small + 메타 프리픽스

- **결정**: text_with_meta 컬럼으로 임베딩 (프리픽스 본문의 ~5%)
- **배경**: 프로젝트 허용 모델. 14,035개 청크 전체 임베딩 비용 207원
- **프리픽스 효과**: `[발주기관: X | 사업명: Y]`를 붙여 키워드 매칭 향상 (Snowflake 연구 5~10%)
- **참조**: 05_embedding.ipynb, 섹션 10

## 7. 벡터 DB: ChromaDB

- **결정**: ChromaDB PersistentClient (not FAISS)
- **이유**: `where={'발주기관': '국민연금공단'}` 같은 메타데이터 필터를 한 줄로 지원. FAISS는 필터링을 직접 구현해야 함.
- **평가 결과**: 필터 적용 시 HR 1.00 vs 순수 벡터 0.78 → **메타 필터가 핵심 성능 요인**
- **참조**: 05_embedding.ipynb, 06_retrieval.ipynb

## 8. 검색: 자동 필터 추출

- **결정**: 질문에서 발주기관명을 자동 감지 → ChromaDB where 필터
- **규칙**:
  - 기관 1개 감지 → 해당 기관 필터
  - 기관 2개+ 감지 → 필터 해제 (다문서 비교)
  - "다른 기관" 감지 → 이전 필터 해제
  - 감지 안 됨 → 대화 맥락에서 이전 필터 유지
- **참조**: 07_generation.ipynb, 섹션 2

## 9. 생성: gpt-5-mini, 프롬프트 v3

- **결정**: 시스템 프롬프트 3회 반복 최적화
- **핵심 지시**:
  - "추측하지 마세요" (할루시네이션 방지)
  - "부분 확인 시 구분" (없는 정보 명시)
  - "다문서 비교: 하나의 표로 간결하게" (토큰 효율)
  - "핵심 먼저, 세부 나중" (일관된 구조)
- **max_completion_tokens**: 8000 (gpt-5 reasoning 토큰 고려, 넉넉하게)
- **temperature**: 미지원 (gpt-5-mini 특성)
- **참조**: 07_generation.ipynb, 섹션 3, 11

## 10. 평가: 5유형 × 4단계

- **결정**: A~E 5가지 질문 유형, 검색→전환→생성→성능 4단계 평가
- **지표**: Hit Rate@5, MRR, nDCG@5, Faithfulness(3단계), Relevance, Context Precision/Recall
- **LLM Judge**: gpt-5-mini, max_completion_tokens=16000 (reasoning 토큰 고려)
- **참조**: 08_evaluation.ipynb
