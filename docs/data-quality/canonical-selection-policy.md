# Canonical Selection Policy

## 목적

원본 문서명과 메타데이터가 서로 다르지만 실제로는 동일한 RFP인 경우,
문서 본문은 하나만 인덱싱하고 메타데이터는 신뢰도 있게 해소하기 위한 기준을 정의한다.

이 문서는 다음 문제를 방지하기 위한 운영 규칙이다.

- 같은 문서가 중복으로 청킹되어 retrieval 점수가 왜곡되는 문제
- CSV 전용 메타데이터를 무심코 버려서 필터링 정보가 사라지는 문제
- 팀원마다 정본 판단 기준이 달라지는 문제

## 적용 범위

이 정책은 다음 조건을 만족하는 중복 후보에 적용한다.

- raw 파일이 2개 이상 존재한다.
- 파일 해시 또는 파싱 본문 비교 결과가 동일하거나 사실상 동일하다.
- 파일명, 발주기관명, 공고 메타데이터가 일부 다르다.

## 기본 원칙

1. 문서 본문은 정본 1개만 인덱싱한다.
2. 중복본은 삭제하지 않고 alias 또는 duplicate 관계로 관리한다.
3. 메타데이터는 문서와 분리해서 해소한다.
4. 정본 선정 근거와 메타데이터 출처는 추적 가능해야 한다.

## 정본 선정 기준

정본은 아래 우선순위에 따라 선택한다.

1. 원문 내부에 명시된 공식 발주기관명과 더 잘 일치하는 파일명을 우선한다.
2. 공고 번호, 공고 차수, 공개 일자, 입찰 참여 시작일, 입찰 참여 마감일이 더 완전한 행을 우선한다.
3. 같은 내용을 담고 있더라도 포털명, 서비스명, 별칭보다 실제 기관명을 포함한 파일명을 우선한다.
4. 위 기준으로도 동률이면 기존 CSV에 포함된 파일명을 우선한다.

## 메타데이터 해소 기준

정본 파일 하나만 남기더라도 중복 행의 메타데이터는 바로 버리지 않는다.

- `resolved_agency`
  - 원문 내부 기관명과 가장 잘 맞는 값을 사용한다.
- `resolved_notice_id`, `resolved_round`, `resolved_dates`
  - 공고 메타데이터가 더 완전한 행의 값을 사용한다.
- `aliases`
  - 다른 파일명, 포털명, 서비스명 표기는 검색용 alias로 보존한다.
- `metadata_sources`
  - 어떤 필드가 어느 행에서 왔는지 추적 가능하게 남긴다.

## 처리 방식

1. `configs/data_quality/duplicates_map.csv`에 중복 그룹과 정본을 기록한다.
2. ingest 단계에서는 `canonical_file`만 파싱, 청킹, 임베딩 대상으로 사용한다.
3. duplicate 행은 skip하되, 메타데이터 해소 단계에서 보강 출처로 사용할 수 있게 유지한다.
4. 결과적으로 retrieval 대상 문서는 1개, metadata 해소 출처는 여러 개가 될 수 있다.

## 현재 확인된 중복 후보

### DUP-001

- `BioIN_의료기기산업 종합정보시스템(정보관리기관) 기능개선 사업(2차).hwp`
- `한국보건산업진흥원_의료기기산업 종합정보시스템(정보관리기관) 기능.hwp`

판단:

- raw 파일 해시가 동일하다.
- 공식 발주기관명 기준으로 `한국보건산업진흥원_...`을 정본으로 본다.
- `BioIN` 표기는 alias로 유지한다.

### DUP-002

- `국가과학기술지식정보서비스_통합정보시스템 고도화 용역.hwp`
- `한국한의학연구원_통합정보시스템 고도화 용역.hwp`

판단:

- raw 파일 해시가 동일하다.
- `한국한의학연구원_...` 쪽이 기관명 기준으로 더 직접적이다.
- 해당 행이 공고 메타데이터도 더 완전하므로 정본으로 사용한다.
- `국가과학기술지식정보서비스` 표기는 alias 및 메타 보강 출처로 유지한다.

### DUP-003

- `고려대학교_차세대 포털·학사 정보시스템 구축사업.docx`
- `고려대학교_차세대 포털·학사 정보시스템 구축사업.pdf`

판단:

- 동일 기관과 동일 사업명으로 확인된다.
- CSV 기준 원본 파일명은 PDF이며 raw 폴더에는 docx가 추가로 존재한다.
- 정본은 `고려대학교_...pdf`로 사용하고 `docx`는 포맷 중복본으로 관리한다.
