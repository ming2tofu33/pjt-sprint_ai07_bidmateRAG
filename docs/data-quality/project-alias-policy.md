# Project Alias Policy

## 목적

사용자가 원문 사업명을 그대로 입력하지 않고 축약형, 약어, 기관+키워드 형태로 질문해도
같은 사업으로 연결될 수 있도록 사업명 alias 사전을 운영하기 위한 기준을 정리한다.

이 문서는 다음 문제를 줄이기 위한 운영 규칙이다.

- 사업명이 너무 길어 질문과 매칭되지 않는 문제
- 약어(`RCMS`, `BIS`, `ISMP`, `AFSIS`)만 입력했을 때 retrieval이 약해지는 문제
- 너무 일반적인 축약어를 넣어서 다른 사업까지 잘못 묶이는 문제

## 관리 파일

- `configs/data_quality/project_alias_map.csv`

## 컬럼 설명

- `canonical_agency`
  - 해당 사업의 대표 기관명
- `canonical_project`
  - 메타데이터 기준 대표 사업명
- `project_alias`
  - 사용자가 실제로 입력할 수 있는 축약형 표현
- `alias_type`
  - 축약 방식 분류
  - 예: `기관+핵심키워드`, `약어`, `핵심축약`
- `source`
  - alias를 만든 출처
  - 예: `manual`, `metadata_title`
- `enabled`
  - 현재 운영 대상 여부
- `note`
  - 충돌 위험, 사용 조건 등 운영 메모

## 기본 원칙

1. 너무 일반적인 표현은 넣지 않는다.
2. 가능하면 기관명 또는 고유 약어가 포함된 alias를 우선한다.
3. 같은 alias가 여러 사업에 걸칠 수 있으면 note에 충돌 위험을 적는다.
4. 초기에는 소수 핵심 사업만 운영하고, 실제 질문 로그를 보며 확장한다.

## 우선 등록 기준

아래 조건 중 하나 이상에 해당하면 1차 등록 후보로 본다.

- 사업명이 매우 길다.
- 약어가 공식 제목에 포함되어 있다.
- eval 질문 또는 실제 사용자 질문에서 축약형으로 자주 등장한다.
- 같은 기관 안에서만 봐도 대표 사업으로 구분이 쉽다.

## 제외 기준

아래 표현은 기본적으로 제외하거나 비활성화 검토 대상으로 둔다.

- `시스템 구축`
- `고도화 사업`
- `운영 용역`
- `홈페이지 구축`

이런 표현은 여러 문서에 동시에 걸릴 가능성이 높다.

## 운영 팁

1. 먼저 `기관+핵심키워드` 형태를 우선 등록한다.
2. 그 다음 공식 약어를 등록한다.
3. 마지막으로 일반 축약형을 넣되 충돌 가능성을 점검한다.
4. 실제 코드 반영 전에는 데이터 파일 수준에서만 검수해도 충분하다.
