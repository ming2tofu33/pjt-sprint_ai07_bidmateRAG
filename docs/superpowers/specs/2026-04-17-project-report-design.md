## 보고서 목차 및 각 섹션 설명

### 1. 표지
- 프로젝트명: **BidMate RAG — 공공입찰 RFP 지식 검색 시스템**
- 팀명, 팀원 5명 + 역할, 제출일

### 2. 프로젝트 개요
- **비즈니스 배경**: 입찰메이트 컨설팅 스타트업, 하루 수백 건의 RFP 수동 검토 문제
- **목표**: 컨설턴트용 사내 RAG Q&A 시스템으로 RFP 핵심 정보 자동 추출
- **범위**: 실제 RFP 100건, 메타데이터 포함
- **시스템 한 줄 요약**: YAML 한 장으로 실험 정의 → CLI 한 줄 실행 → 비용/메트릭 자동 기록 → 마크다운 리포트 생성

### 3. 팀 구성 및 협업 방식
- 팀원별 역할 표
- **Git 브랜치 전략**: `feat/<initial>` 방식, develop 기준선, PR → develop → main 흐름
- **커밋 기여도**: `git shortlog -sn` 통계 삽입
- 주요 PR 목록 (의사결정 토론이 담긴 PR 발췌)

### 4. 시스템 아키텍처
- **전체 파이프라인 다이어그램**: 파싱 → 전처리 → 청킹 → 임베딩 → 검색 → 생성 → 평가 → 리포트
- **기술 스택 표**:

| 구분 | 기술 |
|---|---|
| 파싱 | kordoc (Node.js CLI) |
| 청킹 | LangChain MarkdownHeaderTextSplitter + RecursiveCharacterTextSplitter |
| 임베딩 | text-embedding-3-small (OpenAI) |
| 벡터 DB | ChromaDB (PersistentClient, cosine) |
| LLM | gpt-5-mini (시나리오 B), HF 로컬 (시나리오 A) |
| UI | Streamlit (디버깅) + Next.js (컨설턴트용) |
| 패키지 관리 | Python 3.12 + uv |

### 5. 핵심 기술 의사결정 (10개)
각 결정마다 **문제 → 대안 비교 → 선택 근거 → 결과/수치** 형식으로 서술.
관련 커밋 해시 또는 코드 스니펫을 근거로 첨부.

| # | 단계 | 결정 사항 | 핵심 수치 |
|---|---|---|---|
| D1 | 파싱 | kordoc 선정 | CSV 대비 텍스트량 28배 |
| D2 | 전처리 | 6종 노이즈 제거 | 총 746,144자 제거 (평균 6.7%) |
| D3 | 청킹 전략 | h1 분리 + 500자 병합 | 청크 수 34,983개→적정 수준 |
| D4 | 청크 크기 | 1,000자 + 150자 오버랩 | 한국어 ~0.5토큰/자 기준 |
| D5 | 메타데이터 | 기관유형/사업도메인/기술스택 추가 | 평가셋 대응률 50%→86% |
| D6 | 임베딩 | text-embedding-3-small + 메타 prepend | 검색 5~10% 향상 |
| D7 | 벡터 DB | ChromaDB (vs FAISS) | 메타데이터 필터링 내장 |
| D8 | 검색 | 자동 필터 추출 + 코사인 유사도 | Hit Rate 0.78→1.00 (+22%p) |
| D9 | 생성 | gpt-5-mini + 시스템 프롬프트 v3 | Answer Relevance 100점 |
| D10 | 평가 | 5유형 12문항, Hit Rate/MRR/nDCG + LLM Judge | 4단계 분리 평가 |

### 6. 실험 결과
- **검색 성능 표**: Hit Rate@5 / MRR / nDCG@5 — 실험 단계별 개선 추이
- **생성 성능 표**: Faithfulness / Answer Relevance / Context Precision / Recall
- **주요 비교**: 필터 적용 전후, 청크 크기별, 프롬프트 버전별
- **실험 노트북 참조**: 01~11번 노트북 매핑 표

### 7. UI 및 서비스 구현
- **Streamlit UI**: 3탭 구성 (라이브 데모 / 문서 목록 / 평가), 팀 디버깅용
- **Next.js 웹 UI**: 컨설턴트 실사용 인터페이스
  - 슬래시 커맨드 12개 (`/요약` `/요구사항` `/예산` 등)
  - 문서 미리보기 모달, 전체 카탈로그 모달 (`⌘D`)
  - `@` 문서 멘션, 좌우 2단 근거 패널

### 8. 한계 및 향후 개선 방향
- gpt-5-mini temperature 미지원 → 프롬프트 제어로 대응
- 시나리오 A (로컬 HF/vLLM + PEFT) 확장 계획
- RFP 100건 이외 문서 일반화 가능성
