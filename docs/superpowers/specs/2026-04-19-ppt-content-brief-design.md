# PPT 콘텐츠 브리프 설계 스펙

> 작성일: 2026-04-19
> 대상 산출물: `docs/PPT/PPT_콘텐츠_브리프.md`
> 활용처: Genspark.ai에 투입하여 PPT 슬라이드 초안 자동 생성

---

## 1. 목적 (Purpose)

BidMate RAG 프로젝트의 20분 분량 발표용 슬라이드 30장을 **Genspark.ai가 일괄 생성**할 수 있도록, 슬라이드별 콘텐츠와 지표를 담은 단일 마크다운 브리프 문서를 설계한다.

**해결하려는 문제**: 현재 `docs/PPT/PPT 슬라이드 설계 v2.md`는 **팀 내부 관리용**으로, 발표자 배분·시간 배분·역할 코드(숫자 슬로건 #n 등) 같은 메타 정보가 섞여 있어 Genspark 투입 시 노이즈로 작용한다. 이를 걷어낸 **콘텐츠 전용 브리프**가 별도로 필요하다.

---

## 2. 범위와 제약 (Scope and Constraints)

### 포함 범위
- 슬라이드 번호 + 제목
- 한 줄 Hook 메시지 (있는 경우)
- 핵심 텍스트 (불릿 2~4개)
- 수치/지표/표 (실측 데이터 기반)
- 시각화 제안 (차트 타입 1줄, 예: "막대/라인/깔때기/카드/다이어그램")
- 각주 (샘플 수, 출처, 주의사항)

### 제외 범위 (사용자가 직접 처리)
- 브랜드 컬러·폰트·스타일
- 레이아웃 상세 지시 (위치·간격·정렬)
- 디자인 토큰
- 시각화의 세부 축·색상·강조 지점

### 데이터 베이스라인
- `docs/PPT/PPT 슬라이드 설계 v2.md` (팀 내부 관리용 원본)
- v2의 실측 데이터 출처 표를 그대로 계승 (`bench-*` run_id, `tmp/*.md` 노션 본문 등)

---

## 3. 입력 시나리오 (Input Scenario)

**투입 방식**: **옵션 A — 전체 문서 한 번에 투입**.

Genspark.ai에 30장 분량 단일 마크다운을 통째로 넘기고 "이 브리프대로 PPT 30장을 만들어달라" 요청. 이 시나리오에 맞춰:
- 문서 상단에 **목차 요약표**를 두어 Genspark가 전체 서사를 먼저 파악
- 섹션 그룹핑을 **콘텐츠 흐름 기준**(발표자 이름 아님)으로 정렬
- 각 슬라이드는 동일 템플릿으로 통일하여 파싱 안정성 확보

---

## 4. 문서 구조 (Document Structure)

```
PPT_콘텐츠_브리프.md
├── 1. 발표 개요 (프로젝트·청중·시간·총 장수)
├── 2. 목차 요약표 (30장 전체, 한 줄씩)
└── 3. 콘텐츠 섹션 × 5
    ├── ① 문제·가설·평가 체계 (슬라이드 1~7)
    ├── ② 데이터 엔지니어링 (슬라이드 8~12)
    ├── ③ 청킹·메타데이터 (슬라이드 13~17)
    ├── ④ 검색 아키텍처 (슬라이드 18~23)
    └── ⑤ 생성·시나리오 A·결론 (슬라이드 24~30)
```

섹션 헤더는 `### 섹션 ① 문제·가설·평가 (슬라이드 1~7)` 형식.

---

## 5. 섹션별 세부 설계

### 5.1 발표 개요

문서 맨 위에 **4~6줄**짜리 간결한 메타 블록:

```markdown
# BidMate RAG — 20분 발표 콘텐츠 브리프

- **프로젝트**: 공공입찰 RFP 101건 대상 RAG 질의응답 시스템
- **청중**: 같은 반 동료 + 교수님 (RAG 용어 익숙, RFP 도메인 생소)
- **발표 시간**: 약 20분
- **총 슬라이드**: 30장
- **언어**: 한국어 (기술 용어는 영문 유지)
```

### 5.2 목차 요약표

30장 전체를 플랫 테이블 하나로. Genspark가 전체 서사를 먼저 파악하는 기능 + 발표 슬라이드 2(목차)의 원본 재사용.

**형식**:

```markdown
## 목차 요약표

| # | 제목 | 한 줄 메시지 |
|---|---|---|
| 1 | BidMate RAG + 팀 거버넌스 | RFP 101건 RAG + 5인 Git Worktree 협업 |
| 2 | 목차 | — |
| 3 | 문제: 컨설턴트의 수작업 지옥 | 수백 페이지 RFP 수작업 파악의 비효율 |
| ... | ... | ... |
```

**규칙**:
- "한 줄 메시지"는 청중이 **기억할 한 문장** (제목과 중복 금지)
- **★** 표시로 핵심 슬라이드 강조 (Genspark 가중치 신호)
- **Spine 슬라이드**(8/13/18/24)도 포함 — 실제 장표이므로
- **목차 슬라이드(2)의 메시지는 `—`**로 표기 (기능적 슬라이드)
- 섹션 경계는 시각적 공백 또는 `### 섹션` 헤더로 분리

### 5.3 콘텐츠 섹션 × 5

각 섹션은 `### 섹션 ① 문제·가설·평가 (슬라이드 1~7)` 같은 h3 헤더로 시작하고, 하위에 슬라이드별 엔트리를 표준 포맷으로 나열.

## 6. 슬라이드 표준 포맷 (Per-slide Template)

모든 슬라이드 엔트리는 다음 템플릿으로 통일:

```markdown
### 슬라이드 N — [제목]

> **Hook**: [한 줄 임팩트 메시지]

**핵심 텍스트**:
- [불릿 1]
- [불릿 2]
- [불릿 3]

**수치/표**:
| 지표 | 값 |
|---|---|
| ... | ... |

**시각화 제안**: [차트 타입 1줄]

**각주**: [샘플 수, 출처, 주의사항]
```

### 포맷 규칙

| 규칙 | 이유 |
|---|---|
| Hook·수치표·각주는 없으면 해당 라인 전체 생략 | 빈 섹션 제거 → 문서 간결 |
| **핵심 텍스트는 모든 슬라이드 필수** | Genspark가 없으면 슬라이드 못 만듦 |
| 시각화 제안은 차트 타입 1줄만 ("막대/라인/깔때기/카드/다이어그램") | 색·레이아웃 상세는 사용자 직접 |
| 각주는 수치 신뢰성 정보 (샘플 수·run_id·출처) | 교수 질문 방어용 증거 |
| Placeholder는 `> 📍 Placeholder: [내용]` | Genspark 과생성 방지 |

## 7. 슬라이드 번호 체계

v2 파일 대비 **목차 슬라이드(2) 신설로 +1 shift**:

| 구간 | v2 (29장) | 브리프 (30장) | 내용 |
|---|---|---|---|
| 타이틀 | 1 | 1 | 동일 |
| 목차 | — | **2** 🆕 | 신설 |
| 문제정의~평가셋 품질 | 2~6 | **3~7** | +1 |
| 데이터 | 7~11 | **8~12** | +1 |
| 청킹 | 12~16 | **13~17** | +1 |
| 검색 | 17~22 | **18~23** | +1 |
| 생성·결론 | 23~29 | **24~30** | +1 |

**Visual Spine 위치 조정**:
- 원본 (전체 파이프라인 + KPI): 슬라이드 **5** (v2의 4)
- 재등장 하이라이트: 슬라이드 **8 / 13 / 18 / 24** (v2의 7/12/17/23)

**핵심 슬라이드(★/★★) 위치 조정**:

| 슬라이드 | v2 번호 | 브리프 번호 | 강조도 |
|---|---|---|---|
| 가설 선언 | 3 | **4** | ★ |
| Judge v2 드라마 | 5 | **6** | ★ |
| 평가셋 품질 혁신 | 6 | **7** | ★ |
| Kordoc + Fallback + 유니코드 정규화 | 9 | **10** | ★ |
| 데이터 클리닝 킬러 | 11 | **12** | ★★ |
| chunk_size=1000 결정 | 14 | **15** | ★ |
| 청킹 반전 (단락 > Semantic) | 16 | **17** | ★★ |
| CE 본 경로 분리 3이유 | 19 | **20** | ★ |
| Hard → Soft Boost | 20 | **21** | ★ |
| 깔때기 (13,807 → 5) | 22 | **23** | ★ |
| 프롬프트 소형 모델 벽 (전략적 재정렬) | 25 | **26** | ★ |
| 답변 3섹션 규약 | 26 | **27** | ★ |
| Top-K 실측 + 가설 회수 | 28 | **29** | ★★ |

## 9. Out of Scope (하지 않을 것)

- ❌ HTML/Slidev/reveal.js 구현 — 본 스펙은 콘텐츠 문서만
- ❌ 디자인 토큰 (색상·폰트·간격) — 사용자가 직접
- ❌ 발표자 배분 / 시간 배분 — v2 파일에만 남김 (팀 내부 관리용)
- ❌ Genspark.ai에 투입·실행 — 사용자가 직접
- ❌ 시나리오 A 추가 실험 결과 반영 — 발표 직전에 사용자·시나리오 A 담당자가 직접 Placeholder 갱신

---

## 10. 검증 기준 (Acceptance Criteria)

브리프 파일이 완성되면 다음을 확인:

1. **목차 요약표**에 30장 모두 포함, ★/★★ 표시가 섹션 7의 핵심 슬라이드 매핑 표와 일치
2. 모든 슬라이드 엔트리가 **표준 템플릿**을 따름 (최소한 제목 + 핵심 텍스트)
3. 실측 데이터 출처(run_id, 날짜, 샘플 수)가 **각주에 명시**됨
4. 발표자 이름·시간·역할 코드 **포함되지 않음**
5. 디자인 토큰(색상·폰트) **포함되지 않음**
6. 시나리오 A 관련 슬라이드(25, 29, 30)에 **전략적 재정렬** 메시지 반영됨
7. Placeholder 마커는 `> 📍 Placeholder:` 형식으로 통일
