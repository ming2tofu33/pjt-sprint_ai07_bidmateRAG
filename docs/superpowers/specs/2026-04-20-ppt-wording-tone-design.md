# PPT 슬라이드 문구 톤 정리 설계 스펙

> 작성일: 2026-04-20
> 대상 산출물: `docs/PPT/slide01.html` ~ `docs/PPT/slide30.html`
> 우선 범위: 제목, 훅, 배지 등 시각적으로 가장 먼저 읽히는 문구

---

## 1. 목적 (Purpose)

`docs/PPT/` 아래 발표 슬라이드의 문구를 보다 **학술적이고 단정한 발표 톤**으로 정리한다.

현재 슬라이드는 전체 서사와 데이터 근거가 잘 갖춰져 있지만, 일부 장표의 제목·훅·배지에 다음과 같은 표현이 포함되어 있다.

- 감정 온도가 높은 표현: `드라마`, `클라이맥스`, `킬러 발견`
- 구어적 과장 표현: `한 방에`, `벽을 넘지 못했다`
- 발표용 영어 메타 표현: `Spoiler`, `메타 인사이트`

이 표현들은 슬라이드 자체의 완성도는 높이지만, 발표자가 구두로 읽을 때 학술 발표보다 서사형 피치에 가깝게 들릴 수 있다. 따라서 **강조 포인트는 유지하되 표현의 온도를 낮추는 것**이 이번 작업의 목표다.

---

## 2. 범위와 제약 (Scope and Constraints)

### 포함 범위

- 슬라이드 상단 배지 텍스트
- 메인 제목 (`<title>` 및 본문 대제목)
- 훅 문장 / 부제 / 예고 문구
- 강조 배너나 카드 상단의 섹션성 라벨

### 제외 범위

- 본문 데이터 표, 수치, 차트 값
- 레이아웃, 색상, 도형 구성
- 카드 내부 세부 설명 전체 재작성
- 발표 순서나 서사 구조 자체 변경

### 작업 원칙

- 슬라이드의 **강조 지점은 유지**하되, 감정적 어휘는 축소한다.
- 사용자 기존 작업 흐름을 존중하기 위해 **눈에 띄는 문구부터 우선 수정**한다.
- 이미 충분히 중립적인 슬라이드는 손대지 않는다.

---

## 3. 톤 목표 (Target Tone)

이번 정리는 다음 네 가지 성격을 목표로 한다.

1. **학술적**: 주장보다 관찰·근거·해석 중심으로 표현
2. **단정함**: 과장, 감탄, 도발적 수사를 줄인 문장
3. **명료성**: 제목만 읽어도 무엇을 보여주는 슬라이드인지 파악 가능
4. **발표 적합성**: 발표자가 읽었을 때도 자연스럽고 안정적으로 들리는 문구

### 유지할 요소

- 핵심 숫자 대비 (`MRR 0.000 → 1.000`, `30% → 10%`, `13,807 → 5`)
- 중요 의사결정의 존재감
- 슬라이드 간 수미상관 구조

### 줄일 요소

- 감정적 강조
- 서사적 과장
- 영어 메타 표현
- 한 문장 안의 과도한 대립 구도

---

## 4. 문구 정리 전략 (Strategy)

이번 작업은 **균형형 정리안**을 따른다.

- 서사 구조와 강조 포인트는 유지
- 제목·훅·배지의 직접적 표현만 학술 톤으로 재작성
- 내용의 힘은 수치와 결과에서 나오도록 정리

즉, 슬라이드를 밋밋하게 만드는 것이 아니라 **"보여주는 데이터는 강하게, 설명하는 문구는 절제되게"** 가져간다.

---

## 5. 문구 레이어별 설계

### 5.1 배지 체계

현재 배지는 `★ 임팩트`, `★★ 킬러 발견`, `클라이맥스`, `메타 인사이트` 등 감정형 표현을 포함한다. 이를 **역할 중심 라벨**로 정리한다.

#### 배지 변환 원칙

| 현재 표현 | 목표 표현 |
|---|---|
| `★ 임팩트` | `핵심 결과` |
| `★★ 킬러 발견` | `주요 발견` |
| `★ 의사결정` | `설계 결정` |
| `클라이맥스` | `종합 검토` |
| `메타 인사이트` | `해석` |
| `Spoiler` | `예고` |
| `지금 여기` | `현재 단계` |

#### 추가 원칙

- 슬라이드 본문에서 `★`, `★★` 표시는 제거한다.
- 별표 체계는 내부 기획 문서에서만 유지해도 충분하다.
- 배지는 의미를 설명하는 보조 정보로 남기고, 감정적 연출 수단으로 쓰지 않는다.

### 5.2 제목

제목은 `무엇을 관찰했고 어떤 의미가 있는지`가 드러나도록 바꾼다.

#### 제목 재작성 규칙

- 구어체보다 서술형 명사구를 우선한다.
- `한 방에`, `벽`, `드라마`, `재앙` 같은 자극적 단어는 피한다.
- 필요 시 `배경`, `전환`, `비교`, `변화`, `해석`, `검토`와 같은 중립어로 대체한다.
- 핵심 숫자 대비는 유지하되, 결론형 서술어는 완화한다.

#### 예시

| 현재 제목 | 목표 방향 |
|---|---|
| `Judge v2 드라마 & 구조` | `Judge v2 전환 배경과 평가 구조` |
| `데이터 클리닝 한 방에 청킹 차이가 사라졌다` | `데이터 정제 이후 청킹 전략 간 성능 차이 축소` |
| `프롬프트로는 소형 모델의 벽을 넘지 못했다` | `프롬프트 최적화만으로는 소형 모델 한계 극복이 어려웠다` |
| `Top-K 실측 + 가설 회수` | `Top-K 실측과 가설 검토` |

### 5.3 훅 / 부제 / 예고 문구

훅은 슬로건형 문장보다 **실측 결과의 시사점**을 한 문장으로 정리하는 방식으로 바꾼다.

#### 훅 재작성 규칙

- 직접 인용부호 사용을 최소화한다.
- `반전`, `충격`, `결국`, `문제였다` 같은 단정 표현은 완화한다.
- `실측 결과`, `관찰 결과`, `이 결과는`, `이 슬라이드는` 같은 발표형 연결을 활용한다.
- 한 문장 안에서 메시지를 두 번 강조하지 않는다.

#### 예시

| 현재 훅 | 목표 방향 |
|---|---|
| `측정 도구의 신뢰성이 곧 성능 지표의 신뢰성 — 파이프라인이 아니라 측정 도구가 문제였다` | `실측 결과, 초기 성능 저하는 파이프라인 자체보다 평가 도구의 한계와 더 밀접하게 연결되어 있었다.` |
| `모델/청킹보다 데이터 품질이 RAG 성능의 핵심` | `실험 결과, 데이터 품질 개선이 청킹 전략 변경보다 더 큰 영향을 보였다.` |
| `병목은 시나리오마다 다르다 — 검색은 만점이어도, 모델이 병목이 될 수 있다` | `실측 결과, 병목은 시나리오별로 다르게 나타났으며 검색 성능만으로 전체 품질을 설명하기는 어려웠다.` |

## 6. 우선 수정 대상 (Priority Slides)

전체 슬라이드 중 아래 장표를 1차 우선순위로 본다.

| 슬라이드 | 이유 |
|---|---|
| `slide04.html` | `★ 임팩트`, `Spoiler`, 선언형 문구의 온도가 높음 |
| `slide06.html` | `드라마`, 강한 대립형 훅 사용 |
| `slide12.html` | `★★ 킬러 발견`, `한 방에` 표현 사용 |
| `slide21.html` | 배지에 `드라마` 포함 |
| `slide23.html` | 배지에 `클라이맥스` 포함 |
| `slide26.html` | `벽을 넘지 못했다`, `메타 인사이트` 표현 사용 |
| `slide29.html` | `★★ 클라이맥스`, `가설 회수` 등 서사 강조가 큼 |

필요 시 2차 범위로 다음도 검토한다.

- `slide10.html`, `slide15.html`, `slide20.html`, `slide27.html`
- `지금 여기` 계열 슬라이드 (`slide08.html`, `slide13.html`, `slide18.html`, `slide24.html`)

## 7. 적용 후 기대 상태 (Expected Output)

문구 정리 후 슬라이드는 다음 상태를 만족해야 한다.

- 제목만 읽어도 슬라이드의 역할이 명확하다.
- 발표자가 그대로 읽어도 과장된 느낌이 줄어든다.
- 숫자와 그래프가 메시지의 중심이 되고, 문구는 해석을 보조한다.
- 여러 섹션을 넘나들어도 배지 체계와 문장 톤이 일관된다.

---

## 8. 검증 기준 (Acceptance Criteria)

수정이 완료되면 다음을 확인한다.

1. 우선 수정 대상 슬라이드의 배지가 모두 역할 중심 표현으로 바뀌어 있다.
2. `드라마`, `클라이맥스`, `킬러 발견`, `한 방에`, `벽을 넘지 못했다`, `Spoiler`, `메타 인사이트` 등 고온 표현이 사용자 승인 없이 남아 있지 않다.
3. 제목과 훅이 관찰·해석 중심 문장으로 정리되어 있다.
4. 차트 수치, 표 데이터, 레이아웃에는 변화가 없다.
5. 수정 후에도 발표 흐름의 강조 포인트는 유지된다.

---

## 9. 다음 단계

본 스펙이 승인되면 다음 순서로 진행한다.

1. 우선 수정 대상 슬라이드에 대한 **대체 문구안**을 텍스트로 제시
2. 사용자와 톤을 최종 미세 조정
3. 승인된 문구를 실제 `docs/PPT/slide*.html`에 반영
4. 필요 시 `PPT_콘텐츠_브리프.md`와 표현 일관성도 후속 정리
