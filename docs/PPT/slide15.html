왜 chunk_size = 1000인가

하이브리드 청킹(단락 + 고정크기)에서 RFP 세부 요구사항 1~2개를 온전히 담는 황금비율

small

chunk_size 500
overlap 100
min_section 300
max_table 1,000

적합 ·

팩토이드 단답형 (예: "예산 얼마?")

medium — 채택 (단락 + 고정 하이브리드)

chunk_size 1,000
overlap 150
min_section 500
max_table 1,500

적합 · 범용

요구사항·비교·분석 모두 안정적으로 커버

large

chunk_size 1,500
overlap 200
min_section 700
max_table 2,000

적합 ·

분석·비교형 (예: "A·B 사업 배점 비교")

Header Carry-over: 표 분할 시 헤더 재삽입

max_table 초과 구간을 나눌 때 각 청크의 선두에 열 헤더를 다시 넣어, 검색 결과 단독으로도 컬럼 의미를 유지

원본 표

| 구분 | 점수 | 설명 |
| 기술 | 80 | ... |
| 가격 | 20 | ... |
...

→

청크 A

| 구분 | 점수 | 설명 |
| 기술 | 80 | ... |

청크 B

| 구분 | 점수 | 설명 |
| 품질 | 15 | ... |

효과

각 청크가 열 이름을 함께 유지하므로, 검색 결과만 봐도 "점수·설명"의 의미를 바로 해석 가능

configs/chunking/chunking_1000_150.yaml (채택) · small/medium/large 3종 전부 실측 → medium이 Faithfulness 최고 (슬라이드 17에서 공개)
