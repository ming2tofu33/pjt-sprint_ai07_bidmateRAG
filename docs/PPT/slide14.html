청킹 실패 유형 3가지와 계층형 해법

모델 성능보다 RFP 물리적 구조(표·단락)를 먼저 반영한 청킹 설계

유형 ① 너무 작게

문장 앞뒤 문맥이 절단 → 대명사·지시어 해석 불가

예 ·

"그 사업은 3년간…"
→ "그 사업"의 지시 대상 소실

유형 ② 너무 크게

노이즈 함유량 증가 → 검색 정밀도 하락

예 ·

2,000자 청크 = 사업개요
+ 요구사항 + 일정 + 예산 ...
→ 핵심 신호 희석

유형 ③ 경계 무시

논리 경계 무시 → 두 요구사항이 한 청크에 섞임

예 ·

"보안 요구 | 성능 요구"
한 청크에 공존
→ 분리 불가, 검색 오염

설계 방향 · 하이브리드 청킹 = 단락 기반(1단) + 고정크기(2단) · Semantic(3단)은 비교 후보

1단 · 단락 Markdown 헤더 분할

Kordoc 출력의 # 헤더 기준 1차 분할

# 사업개요
## 요구사항 · 보안
## 요구사항 · 성능

min_section_size = 500자 (작은 섹션 병합)

2단 · 고정크기 Recursive Character

큰 섹션을 문자 수 기준 재귀 분할

separators = [
  "\n\n", "\n", ". ", " "
]

자연 경계 우선, 점진적으로 세분화

3단 · 대안 Semantic Chunker

임베딩 유사도 급변 지점에서 주제 경계 감지

breakpoint_type =
  "percentile", amount = 95
