#### 슬라이드 5 · 파이프라인 & 정직한 KPI 5축

**핵심**: 5개 KPI 정의 + A/B 시나리오 공동 측정

**발표 (45초)**:
> 평가 체계는 **5개 KPI**로 정의했습니다. MRR(검색 순위), Faithfulness(환각 방어), **4지표 평균 — Faithfulness·Answer Relevance·Context Precision·Context Recall로 구성된 RAGAS 표준 평균**, Latency, Cost/query입니다. 두 시나리오 — **보안 요건 대응 로컬 모델 시나리오 A(Gemma-4-E4B)**와 **품질 기준 상용 API 시나리오 B(GPT-5-mini)** — 를 동일 조건으로 비교했고, Latency만 체급 차를 반영해 **A는 90초, B는 30초**의 차등 목표를 설정했습니다. 5개 중 4개는 양쪽 달성, Latency만 B 근접 미달입니다.

#### 슬라이드 6 · MRR 0.000 → 1.000: Judge v2 전환 배경

**핵심**: Judge v1 버그 → v2 재설계로 평가 엔진 신뢰성 확보

**발표 (40초)**:
> 초기 4회 연속 벤치마크에서 **MRR이 0.000**이 나왔습니다. 파이프라인 실패가 아니라, Judge v1의 단순 문자열 매칭 로직이 저희 **3섹션 구조화 답변 포맷 — 핵심·근거·없는 항목, 뒤 슬라이드 27에서 자세히** — 을 못 파싱한 것이었습니다. **Judge v2를 evidence-based ratio scoring**으로 재설계했더니 즉시 MRR 1.000으로 정상화됐습니다. 이 사례가 시사하는 점은 **평가 엔진 신뢰성 없이는 성능 지표도 믿을 수 없다**는 것입니다.

#### 슬라이드 15 · 왜 chunk_size = 1000인가

**핵심**: 500/1000/1500/Semantic 비교 → 1000이 Faith 최고

**발표 (35초)**:
> chunk_size 실험 결과, **1000자·overlap 150자**를 베이스라인으로 채택했습니다. 500자는 문맥 부족, 1500자는 검색 정밀도 하락, Semantic Percentile은 대형 모델 비용 대비 개선 효과가 작았습니다. 1000/150 조합이 청크 수 7,943개, 테이블 비율 14.2%로 가장 균형 잡힌 결과를 보였습니다.

#### 슬라이드 19 · Naïve Dense 검색의 3대 실패 → Hybrid 해결

**핵심**: 기관혼동·고유명사 실패·대명사 — Hybrid+BM25+멀티턴으로 해결

**발표 (45초)**:
> Dense 임베딩 단독으로 검색했을 때 **3대 실패 패턴**이 관찰됐습니다: ① 유사 맥락의 **타 기관 문서가 혼입**, ② **숫자·고유명사** 검색 실패(예: 오타·특수문자), ③ **대명사**가 섞인 후속 질문에서 맥락 소실. 이 세 실패를 각각 **메타 필터 + Hybrid, Sparse BM25, 멀티턴 쿼리 재작성**으로 해결했고, **Dense(의미 유사도) + Sparse BM25(키워드 정확 매칭) + RRF(Reciprocal Rank Fusion, 두 순위 융합) 하이브리드**를 최종 채택했습니다.

#### 슬라이드 20 · Cross-Encoder를 본 경로에서 분리한 3가지 이유

**핵심**: CE 배제 결단 — VRAM 제약 · 기여도 미미 · 공정 A/B

**발표 (45초)**:
> 초기에는 Cross-Encoder(리랭커)를 도입했다가 **최종적으로 본 경로에서 분리**했습니다. 세 가지 이유입니다. ① **A 시나리오의 인프라인 L4 GPU 24GB VRAM**에서 Gemma-4-E4B와 CE 동시 로드 불가, ② Hybrid+RRF 이후 CE 기여도가 오히려 Faith **-0.05 하락**, ③ A/B 공정 비교 원칙. 이는 **데이터 기반 의사결정**의 대표 사례이며, MRR 1.0 도약의 엔지니어링 결단 중 하나입니다.

**핵심**: Soft Boost = 가중치 합산 (상한 0.15), 리랭커 아님

**발표 (45초)**:
> 초기에는 Hard Filter를 썼지만, LLM 섹션 추론이 틀리면 **정답 청크가 완전 배제**되는 치명적 결함이 있었습니다. 이를 **Soft Boost**로 전환 — 섹션·테이블·메타데이터 가중치를 Hybrid 점수에 **상한 0.15 내에서 가산**하는 방식입니다. **리랭커가 아닌 스코어링 보조**이며, 정답 차단 위험 없이 정답 순위만 올립니다.

#### 슬라이드 22 · 멀티턴 재작성 안전장치

**발표 (30초)**:
> 멀티턴 재작성에서 **수치·금액·일정**을 프롬프트에 포함하면 검색이 잘못 좁혀지는 문제가 있었습니다. 안전 슬롯(기관명·사업명·속성)만 재작성에 포함하고, 위험 슬롯(숫자 정보)은 제외하는 **분리 설계**로 해결했습니다.

#### 슬라이드 23 · 7,943 → 5: 6단계 검색 축소 파이프라인 ⭐

**핵심**: 메타 필터 → Dense → Sparse → RRF → Soft Boost → Top-K

**발표 (50초)**:

#### 슬라이드 26 · 프롬프트 최적화 효과가 모델 간 차이보다 크게 나타났다 ⭐

**핵심**: 프롬프트 편차 0.33 > 모델 간 차이 <0.1 → 상호 보완 프레임

**발표 (50초)**:
> 이 섹션의 핵심 발견입니다. 20종 실험에서 **프롬프트만으로 0.5156 → 0.8438을 달성**했습니다. 즉 프롬프트 최적화 편차 **0.33이 모델 간 차이 추정 0.1의 3배 이상**입니다. 결론은 "**A와 B는 위계가 아니라 상호 보완**" — **A는 비용 우위 (per-query 16.2× 저렴, 로컬 인프라), B는 품질·속도 우위 (Faith 1.00, 2.7× 빠름)**. 5원칙을 도출했습니다: 한국어 통일, 1000자 최적, 예시 3개, 영어 CoT 활용, P4 페르소나.

#### 슬라이드 27 · 답변 3섹션 규약: 핵심·근거·문서에 없는 항목

#### 슬라이드 29 · Top-K 실측과 가설 검토

**핵심**: 검색 1.0 수렴, 강점 분화 — 상호 보완 서사
